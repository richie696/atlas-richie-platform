# 一致性联调清单与配置模板

本文用于落地“全局异步 + 关键步骤同步”的一致性方案，覆盖订单/支付的推荐配置与联调检查项。

## 1. 推荐配置模板

### 1.1 应用级配置（application.yml）

```yaml
platform:
  component:
    statemachine:
      enabled: true
      # 全局默认：异步持久化
      db-persistence-mode: ASYNC
      # 严格模式：禁止全局 SYNC 被状态/步骤降级为 ASYNC
      strict-persistence-mode: true
      # 同步持久化失败重试
      sync-retry-times: 2
      sync-retry-interval-ms: 100
```

### 1.2 支付状态机（建议全局同步）

```yaml
name: payment
description: 支付状态机
dbPersistenceMode: SYNC

states:
  - name: WAIT_PAY
    type: INITIAL
  - name: PAYING
    type: NORMAL
  - name: PAY_SUCCESS
    type: FINAL
  - name: PAY_FAIL
    type: ERROR
```

### 1.3 订单状态机（建议全局异步，关键状态同步）

```yaml
name: order
description: 订单状态机
dbPersistenceMode: ASYNC

states:
  - name: CREATED
    type: INITIAL
  - name: WAIT_PAY
    type: NORMAL
  - name: PAID
    type: NORMAL
    statePersistenceMode: SYNC
  - name: CLOSED
    type: FINAL
    statePersistenceMode: SYNC
```

### 1.4 步骤级覆盖（优先级最高）

```yaml
transitions:
  - name: pay_success
    fromState: WAIT_PAY
    toState: PAID
    event: PAY_SUCCESS
    attributes:
      statePersistenceMode: SYNC
```

优先级：`transition > state > machine > global default`。

## 2. 联调测试清单

### 2.1 基础正确性

- 支付机 `SYNC`：迁转成功返回后，数据库已能查到最新状态。
- 订单机 `ASYNC`：迁转成功后，Redis 先更新，数据库随后可见。
- 关键状态覆盖 `SYNC`：仅指定状态/步骤同步，其他仍异步。

### 2.2 冲突与校验

- 全局 `SYNC` + 状态/步骤 `ASYNC`：严格模式下启动失败。
- 关闭严格模式后同样配置：运行期应记录告警并强制按 `SYNC` 执行。
- 检查日志是否包含 `machineMode/stateMode/stepMode/effectiveMode`。

### 2.3 乱序与幂等

- 同一 `stateMachine + businessId` 连续触发多次迁转，验证 `seq` 单调递增。
- 人工重放旧消息，验证旧 `seq` 不覆盖新状态。
- 重复回调（同业务语义）验证幂等，不产生错误回退。

### 2.4 回调先到场景（重点）

- 构造支付回调早于订单异步落库窗口到达：
  - 支付状态机应成功迁转。
  - 订单关键步骤若为 `SYNC`，应避免“回调到达但订单态未落库”的失败窗口。

### 2.5 长尾兜底

- 人为制造同步持久化异常，验证有限重试生效。
- 超过重试后验证死信记录写入。
- 验证巡检日志能发现死信积压并告警。

## 3. 发布与灰度建议

- 灰度顺序：先支付机，再订单关键链路。
- 观测指标：
  - 迁转接口 P95/P99
  - 数据库 QPS / 慢 SQL / 连接池等待
  - 死信积压数量、重试成功率
- 回滚策略：异常波动时可快速切回 `ASYNC`（仅支付强一致链路保留 `SYNC`）。
