# 多状态机使用指南

## 概述

- Redis 是单一事实源（Single Source of Truth）：当前状态与历史均先写入 Redis。
- 事件仅作为触发器：`StateMachineEngine.fire(...)` 触发状态变更，并把同步键投递到 Redis Stream。
- 异步持久化：`StateSyncStreamConsumer` 批量从 Redis 读取最新状态/历史并落库（MyBatis Plus + Liquibase）。
- 表结构版本化：Liquibase 维护 `statemachine_state_current` 与 `statemachine_state_history`。
- 清理机制：定时使用 `XTRIM MAXLEN ~` 清理 Redis Stream（支持全局/按消费者配置的 maxLen，服务级别分布式锁）。

## 快速开始

### 1) 引入依赖

```xml
<dependency>
    <groupId>com.richie.component</groupId>
    <artifactId>atlas-richie-component-statemachine</artifactId>
</dependency>

<!-- DAO 组件（提供 MyBatis Plus 基础能力） -->
<dependency>
    <groupId>com.richie.component</groupId>
    <artifactId>atlas-richie-component-dao</artifactId>
</dependency>
```

### 2) 基础配置（示例）
```yaml
platform:
  component:
    statemachine:
      storage:
        db:
          enabled: true
          batch-size: 100
      schema:
        enable-liquibase: true
        liquibase-change-log: classpath:db/changelog/statemachine/db.changelog-master.yaml
        liquibase-dry-run: false

cache:
  redis:
    stream:
      consumers:
        enabled: true
        configs:
          state-sync:
            stream-key: platform:statemachine:db:sync
            group: statemachine-db-sync-group
            consumer: ${HOSTNAME:statemachine-db-sync-consumer}
            target-type: event.com.richie.component.statemachine.StateSyncMessage
            count: 50            # 单次拉取
            concurrency: ${STATEMACHINE_DB_SYNC_CONCURRENCY:8}  # 消费并发（一般=CPU核心数）
            auto-ack: true
            error-strategy: SKIP
            idempotency-enabled: true
            auto-start: true
      cleanup:
        enabled: true
        interval: 1h
        default-max-len: 3000
```

## 定义状态机

- 配置位置：默认扫描 `classpath:statemachine/**/*.yml|yaml|json`（可通过 `platform.component.statemachine.state-machine.config-path` 调整）。
- 建议：YAML/JSON 配置“拓扑模型”（名称、状态、转换、条件、动作），业务定义枚举提供编译期约束（名称需与 YAML 一致）。

最小示例：
```yaml
# src/main/resources/statemachine/payment-statemachine.yml
name: payment
description: 支付状态机

states:
  - name: PENDING
    description: 待支付
    type: INITIAL

  - name: PROCESSING
    description: 支付中
    type: NORMAL

  - name: SUCCESS
    description: 支付成功
    type: FINAL

  - name: FAILED
    description: 支付失败
    type: ERROR

transitions:
  - name: start_payment
    description: 开始支付
    fromState: PENDING
    toState: PROCESSING
    event: START_PAYMENT

  - name: payment_success
    description: 支付成功
    fromState: PROCESSING
    toState: SUCCESS
    event: PAYMENT_SUCCESS

  - name: payment_failed
    description: 支付失败
    fromState: PROCESSING
    toState: FAILED
    event: PAYMENT_FAILED
```

枚举建议（与 YAML 中名称一致）：
```java
public enum PaymentEvent { START_PAYMENT, PAYMENT_SUCCESS, PAYMENT_FAILED }
public enum PaymentState { PENDING, PROCESSING, SUCCESS, FAILED }
```

## 在业务代码中使用

### 枚举（必须与 YAML/JSON 名称一致）

- 目的：提供编译期约束，避免字符串拼写错误；同时便于 IDE 自动完成。
- 要求：枚举名称需与 YAML/JSON 中的 `states[].name` 与 `transitions[].event` 完全一致；状态机名称也建议使用枚举以匹配静态外观签名。

```java
// 状态机名称（建议使用枚举以匹配 StateMachine 静态门面签名）
public enum PaymentSm { payment }

public enum PaymentEvent { START_PAYMENT, PAYMENT_SUCCESS, PAYMENT_FAILED, REOPEN }
public enum PaymentState { PENDING, PROCESSING, SUCCESS, FAILED, REOPENED }
```

### 使用静态外观类 StateMachine（推荐）

```java
import com.richie.component.statemachine.StateMachine;

// 1) 触发状态变更（基础版）
StateMachine.fire(PaymentSm.payment, PaymentEvent.START_PAYMENT, orderId);

// 2) 触发状态变更（带上下文属性）
Map<String, Object> attrs = Map.of("operator", userId, "channel", "WEB");
StateMachine.

fire(PaymentSm.payment, PaymentEvent.PAYMENT_SUCCESS, orderId, attrs);

// 3) 触发状态变更（显式传入当前状态，避免一次读取）
StateMachine.

fire(PaymentSm.payment, PaymentEvent.PAYMENT_FAILED, orderId, PaymentState.PROCESSING);

// 4) 触发状态变更（同时具备上下文与当前状态，低延迟路径）
StateMachine.

fire(PaymentSm.payment, PaymentEvent.REOPEN, orderId, attrs, PaymentState.SUCCESS);

// 5) 获取当前状态（字符串）
String current = StateMachine.getCurrentState(PaymentSm.payment, orderId);

// 6) 获取当前状态（枚举）
PaymentState currEnum = StateMachine.getCurrentState(PaymentSm.payment, orderId, PaymentState.class);

// 7) 能否触发某事件（按钮可用/前置校验）
boolean can = StateMachine.canTransitionTo(PaymentSm.payment, PaymentEvent.PAYMENT_SUCCESS, orderId);

// 8) 获取状态历史（审计/追踪）
List<StateHistory> history = StateMachine.getStateHistory(PaymentSm.payment, orderId);
```

使用时机建议：
- 使用 1)：简单触发，不关心上下文，读一次当前状态可接受。
- 使用 2)：条件/动作依赖上下文（操作者、原因、来源渠道等）。
- 使用 3)：业务方已持有当前状态，需规避一次读开销（如内存缓存命中）。
- 使用 4)：对延迟敏感，且既需要上下文又已知当前状态。
- `getCurrentState`：展示/判断当前状态，或作为多状态联合检索的输入。
- `canTransitionTo`：前端按钮可点击态、服务端幂等校验/灰度放行前置判断。
- `getStateHistory`：问题排查、用户可视化时间线、审计导出。

### 条件表达式与动作表达式（SpEL）使用指南

SpEL 用于解析 YAML/JSON 中每条 `transition` 的 `condition` 与 `action`。表达式可使用以下变量（由 `StateContext` 注入）：

- `context`：完整上下文对象（可读写 `attributes`）
- `currentState`：当前状态字符串
- `previousState`：前一状态字符串
- `event`：当前触发的事件名
- 通过 `attributes` 注入的键值对（如 `operator`、`channel`、`amount` 等）

示例变量注入（业务触发时传入 attributes）：
```java
Map<String, Object> attrs = Map.of(
    "operator", userId,
    "operatorRole", role,
    "channel", "WEB",
    "amount", orderAmount
);
StateMachine.fire(PaymentSm.payment, PaymentEvent.PAYMENT_SUCCESS, orderId, attrs);
```

#### 常见条件表达式示例（condition）

```yaml
transitions:
  - name: pay_success_admin_only
    fromState: PROCESSING
    toState: SUCCESS
    event: PAYMENT_SUCCESS
    # 仅管理员或金额小于等于 500 的场景允许成功
    condition: "operatorRole == 'ADMIN' || amount <= 500"

  - name: reopen_window
    fromState: SUCCESS
    toState: REOPENED
    event: REOPEN
    # 完成后 24 小时内允许重开（SpEL沙箱禁止方法调用，需在业务代码中预计算）
    condition: "context.attributes['elapsedHours'] != null and context.attributes['elapsedHours'] <= 24"

  - name: guarded_transition
    fromState: PENDING
    toState: PROCESSING
    event: START_PAYMENT
    # 需要渠道为 WEB 且金额>0
    condition: "channel == 'WEB' && amount > 0"
```

说明：表达式返回 `true` 才会放行；表达式异常按不满足处理（视为 `false`）。

#### 常见动作表达式示例（action）

动作在条件通过后执行，适合设置上下文标记、采集审计信息、轻量副作用（不建议在表达式中做外部 IO）。

```yaml
transitions:
  - name: stamp_complete_time
    fromState: PROCESSING
    toState: SUCCESS
    event: PAYMENT_SUCCESS
    condition: "amount > 0"
    # 将业务完成时间放入上下文，消费者可据此写入 updated_at
    action: "context.attributes['completedAt'] = '2025-01-01T00:00:00'"

  - name: mark_reason
    fromState: PROCESSING
    toState: FAILED
    event: PAYMENT_FAILED
    action: "context.attributes['failReason'] = 'GATEWAY_TIMEOUT'"

  - name: enrich_audit
    fromState: PENDING
    toState: PROCESSING
    event: START_PAYMENT
    action: "context.attributes['audit'] = 'start by ' + #event"
```

#### 失败行为与幂等说明

- 条件为 `false` 或执行异常：不触发转换，返回失败结果；不会写入 `current/history`、不会发布 Spring 事件与 Redis 同步消息。
- 无匹配转换：直接失败（"没有找到匹配的转换规则"）。
- 成功路径：引擎会写入当前状态，按需写入历史与发布事件/同步消息；`updated_at` 由消费者按历史业务时间刷新。

#### 最佳实践

- 表达式只做“轻逻辑”：复杂校验与外部依赖下沉到代码，表达式聚焦布尔门禁与轻量动作。
- 避免 IO：动作里不要做 RPC/DB/网络访问；可设置 `context.attributes` 供后续链路使用。
- 显式可读：优先使用简单可读的条件；多条件用括号明确优先级。
- 安全输入：仅暴露可信数据到 `attributes`；不要把原始外部输入直接暴露给表达式。

## 持久化与一致性
- 引擎在状态变更成功后，发布 `StateSyncMessage(syncKey)` 到 Redis Stream（只包含同步键，如 `payment:ORDER123`）。
- 消费者从 Redis 读取“当前最新状态 + 历史”，按批次落库：
  - 当前表：`statemachine_state_current(state_machine, business_id, current_state, updated_at)`
  - 历史表：`statemachine_state_history(id, state_machine, business_id, prev_state, curr_state, event_name, occurred_at)`
- `updated_at` 采用业务含义时间：取该业务键最新一条历史记录的 `occurred_at`；若不存在历史则回退到 `LocalDateTime.now()`。

## 完结状态不可变更（支持白名单重开）

- 原则
  - 类型为 `FINAL` 或 `ERROR` 的状态视为"完结状态"，默认不再允许普通转换离开。
  - 仅当显式声明"重开/撤销"类转换时，才允许从完结状态跳转（白名单）。
  - 在转换定义中设置 `attributes.reopen: true` 即可允许从 FINAL/ERROR 状态转换。

- YAML 约定（示例）

**示例一：支付退款场景**
```yaml
name: payment
states:
  - name: SUCCESS
    type: FINAL
  - name: FAILED
    type: ERROR
  - name: REFUNDING
    type: NORMAL
  - name: REFUNDED
    type: FINAL

transitions:
  # 正常流转到终态
  - name: payment_success
    fromState: PROCESSING
    toState: SUCCESS
    event: PAYMENT_SUCCESS

  # 显式重开（白名单转换）：从支付成功状态开始退款
  - name: start_refund
    fromState: SUCCESS
    toState: REFUNDING
    event: START_REFUND
    condition: "context.attributes['canRefund'] != null and context.attributes['canRefund'] == true"
    # 关键：设置 attributes.reopen: true 允许从终态转换
    attributes:
      reopen: true
    action: "context.attributes['refundStartTime'] = '2025-01-01T00:00:00'"
```

**示例二：订单退单场景（已完成订单退单）**
```yaml
name: order
states:
  - name: PENDING
    type: INITIAL
  - name: CONFIRMED
    type: NORMAL
  - name: COMPLETED
    type: FINAL      # 已完成状态（终态）
  - name: CLOSED
    type: FINAL      # 订单关闭状态（退单后的终态）

transitions:
  # 正常流转到完成状态
  - name: complete
    description: 完成订单
    fromState: CONFIRMED
    toState: COMPLETED
    event: COMPLETE
    action: "context.attributes['completedTime'] = '2025-01-01T00:00:00'"

  # 已完成订单退单：需要设置 reopen: true 允许从终态转换
  - name: refund_from_completed
    description: 已完成订单退单
    fromState: COMPLETED  # FINAL 状态
    toState: CLOSED
    event: REFUND
    # 关键：设置 attributes.reopen: true 允许从终态转换
    attributes:
      reopen: true
    # 可选：添加条件判断，比如只能在一定时间内退单
    condition: "context.attributes['canRefund'] != null and context.attributes['canRefund'] == true"
    action: "context.attributes['refundTime'] = '2025-01-01T00:00:00'; context.attributes['refundReason'] = #event"
```

**使用方式（订单退单示例）**
```java
@Service
public class OrderService {
    
    /**
     * 已完成订单退单
     */
    public void refundCompletedOrder(String orderId, boolean canRefund, String reason) {
        Map<String, Object> attributes = new HashMap<>();
        attributes.put("canRefund", canRefund);
        attributes.put("reason", reason);
        attributes.put("operator", "admin");
        
        StateTransitionResult result = StateMachine.fire(
            OrderSm.order,      // 状态机名称
            OrderEvent.REFUND,   // 退单事件
            orderId,             // 订单ID
            attributes           // 上下文属性
        );
        
        if (result.isSuccess()) {
            log.info("订单退单成功: {}", orderId);
        } else {
            throw new RuntimeException("订单退单失败: " + result.getErrorMessage());
        }
    }
}
```

- 关键点说明
  - `attributes.reopen: true` **必须在转换定义中设置**，而不是在调用时传入。
  - 该设置允许从 FINAL/ERROR 状态转换，是**白名单机制**。
  - 可配合条件表达式，控制何时允许退单（如时间限制、权限检查等）。
  - 引擎会检查：当前状态是否为 FINAL/ERROR，匹配的转换是否设置了 `attributes.reopen: true`，只有设置了 `reopen: true` 的转换才会被允许执行。

- 引擎行为
  - 变更前校验：若当前为完结状态且本次事件未标记为 `attributes.reopen: true`，则直接拒绝（fail fast）。
  - 一致性保障：在写入 Redis 前做原子校验（如 Lua/CAS），避免并发覆写完结状态。
  - 审计：拒绝的事件可记录到日志或历史，便于定位误触。

## 命名与唯一性约定（state_machine 与 business_id）

- 命名空间化推荐
  - 使用有语义的全局唯一 `state_machine` 名称以区分业务与维度：
    - 订单相关：`order.payment`、`order.fulfillment`、`order.delivery`
    - 支付信息表：`payinfo.payment`
  - 这样 `(state_machine, business_id)` 足以唯一定位同一业务 ID 在不同维度下的状态。

- 示例：命名空间化的多状态机 YAML
```yaml
# src/main/resources/statemachine/order-payment.yml
name: order.payment
states:
  - { name: PENDING, type: INITIAL }
  - { name: SUCCESS, type: FINAL }
transitions:
  - { name: start, fromState: PENDING, toState: SUCCESS, event: START }

# src/main/resources/statemachine/order-fulfillment.yml
name: order.fulfillment
states:
  - { name: PENDING, type: INITIAL }
  - { name: READY, type: NORMAL }
  - { name: COMPLETED, type: FINAL }
transitions:
  - { name: confirm, fromState: PENDING, toState: READY, event: CONFIRM }
  - { name: complete, fromState: READY, toState: COMPLETED, event: COMPLETE }

# src/main/resources/statemachine/payinfo-payment.yml
name: payinfo.payment
states:
  - { name: PENDING, type: INITIAL }
  - { name: SUCCESS, type: FINAL }
transitions:
  - { name: pay_ok, fromState: PENDING, toState: SUCCESS, event: PAYMENT_SUCCESS }
```

- 若无法保证命名唯一（可选扩展）
  - 扩展列方案（需要修改表与唯一约束）：新增 `aggregate_type`/`business_type`，主键改为 `(state_machine, aggregate_type, business_id)`；历史表的唯一约束同步扩展。

- 示例：可选扩表示意（DDL 方向，非默认）
```yaml
# 仅示意：如需扩展，请在自有模块维护迁移
- addColumn:
    tableName: statemachine_state_current
    columns:
      - column:
          name: aggregate_type
          type: varchar(64)
          remarks: 业务聚合类型（如 order、payinfo）
- dropPrimaryKey:
    tableName: statemachine_state_current
- addPrimaryKey:
    tableName: statemachine_state_current
    columnNames: state_machine, aggregate_type, business_id
    constraintName: pk_statemachine_state_current_ext
```

## 表结构与注释（由 Liquibase 维护）
- `statemachine_state_current`
  - 复合主键：(state_machine, business_id)
  - `updated_at` 默认 `CURRENT_TIMESTAMP`，更新时按业务发生时间覆盖。
- `statemachine_state_history`
  - 唯一约束：(state_machine, business_id, occurred_at)
  - 常用索引：业务维度与时间维度（见 `003-indexes.yaml`）。

## Redis Stream 运维
- 批量拉取：`count` 可配置；并发：`concurrency` 建议=CPU核心数。
- 清理：
  - 全局 `cleanup.default-max-len`；或在消费者配置 `max-len` 覆盖。
  - 使用 `XTRIM MAXLEN ~` 近似清理，性能友好；无需长度预检。
  - 服务级分布式锁：同一服务的多个实例仅一个执行清理，不同服务互不影响。

## 筛选与分页（订单场景）

### 单状态筛选与分页（最常见）

- 需求：按某一状态机维度（如 `order.payment`）的状态与更新时间过滤订单，并分页返回。
- 推荐索引：`statemachine_state_current(state_machine, current_state, updated_at, business_id)`

写法一：JOIN（直观）
```sql
SELECT  o.*
FROM    orders o
JOIN    statemachine_state_current s
        ON s.business_id = o.order_id
       AND s.state_machine = 'order.payment'
       AND s.current_state IN ('SUCCESS')
WHERE   s.updated_at >= :from
ORDER BY o.created_at DESC
LIMIT :limit OFFSET :offset;
```

写法二：EXISTS（大表友好）
```sql
SELECT  o.*
FROM    orders o
WHERE   EXISTS (
          SELECT 1 FROM statemachine_state_current s
          WHERE s.business_id = o.order_id
            AND s.state_machine = 'order.payment'
            AND s.current_state IN ('SUCCESS')
            AND s.updated_at >= :from
        )
ORDER BY o.created_at DESC
LIMIT :limit OFFSET :offset;
```

写法三：两步法（先取ID，再回表；分页最稳）
```sql
-- Step 1: 先在状态表分页取 business_id
SELECT s.business_id
FROM   statemachine_state_current s
WHERE  s.state_machine = 'order.payment'
  AND  s.current_state IN ('SUCCESS')
  AND  s.updated_at >= :from
ORDER BY s.updated_at DESC
LIMIT :limit OFFSET :offset;

-- Step 2: 回表订单
SELECT o.*
FROM   orders o
WHERE  o.order_id IN (:businessIds)
ORDER BY o.created_at DESC;
```

提示：MyBatis Plus 可直接两步实现：Mapper A 查 `business_id` 列表，Mapper B 用 `IN` 分页回表；或用 XML 自定义 JOIN/EXISTS。

### 按多状态筛选与分页

- 业务需求：按两个（或多个）状态机维度过滤订单，并结合状态时间或订单时间分页返回。
- 推荐索引（见 `003-indexes.yaml`）：在 `statemachine_state_current` 建立复合索引
  - `(state_machine, current_state, updated_at, business_id)`
  - 仅按时间筛选时可用 `(state_machine, updated_at, business_id)`

### 方案一：自连接 current 表（直观）

```sql
SELECT  o.*
FROM    orders o
JOIN    statemachine_state_current s_pay
        ON s_pay.business_id = o.order_id
       AND s_pay.state_machine = 'order.payment'
       AND s_pay.current_state IN ('SUCCESS')
JOIN    statemachine_state_current s_fulfill
        ON s_fulfill.business_id = o.order_id
       AND s_fulfill.state_machine = 'order.fulfillment'
       AND s_fulfill.current_state IN ('READY','COMPLETED')
WHERE   s_pay.updated_at     >= :pay_from
  AND   s_fulfill.updated_at >= :fulfill_from
ORDER BY o.created_at DESC
LIMIT :limit OFFSET :offset;
```

适用：状态维度较少；需要一次性回表并排序。注意使用上面的复合索引避免放大扫描。

### 方案二：双 EXISTS（避免结果集放大）

```sql
SELECT  o.*
FROM    orders o
WHERE   EXISTS (
          SELECT 1 FROM statemachine_state_current s
          WHERE s.business_id = o.order_id
            AND s.state_machine = 'order.payment'
            AND s.current_state IN ('SUCCESS')
            AND s.updated_at >= :pay_from
        )
  AND   EXISTS (
          SELECT 1 FROM statemachine_state_current s
          WHERE s.business_id = o.order_id
            AND s.state_machine = 'order.fulfillment'
            AND s.current_state IN ('READY','COMPLETED')
            AND s.updated_at >= :fulfill_from
        )
ORDER BY o.created_at DESC
LIMIT :limit OFFSET :offset;
```

适用：订单表大、单条匹配率低；EXISTS 可良好利用索引并短路。

### 方案三：两步法（先取ID交集，再回表；分页最稳定）

```sql
-- Step 1: 取符合两个状态条件的业务ID（仅取一页）
WITH ids AS (
  SELECT s1.business_id
  FROM   statemachine_state_current s1
  JOIN   statemachine_state_current s2
         ON s2.business_id = s1.business_id
        AND s2.state_machine = 'order.fulfillment'
        AND s2.current_state IN ('READY','COMPLETED')
        AND s2.updated_at >= :fulfill_from
  WHERE  s1.state_machine = 'order.payment'
    AND  s1.current_state IN ('SUCCESS')
    AND  s1.updated_at >= :pay_from
  ORDER BY GREATEST(s1.updated_at, s2.updated_at) DESC
  LIMIT :limit OFFSET :offset
)
SELECT o.*
FROM   orders o
JOIN   ids ON ids.business_id = o.order_id
ORDER BY o.created_at DESC;
```

适用：需要严格的跨表时间排序或减少一次 JOIN 的列传输；亦便于应用层用 MyBatis Plus 先查 `business_id` 列表，再分批回表。

### 索引与实现提示

- `statemachine_state_current`
  - 建议索引名：`idx_current_sm_state_updated_bid`（state_machine, current_state, updated_at, business_id）
  - 备选：`idx_current_sm_updated_bid`（state_machine, updated_at, business_id）
- `statemachine_state_history`（如需时间回溯查询）
  - `idx_history_sm_occurred_bid`（state_machine, occurred_at, business_id）
- MyBatis Plus：
  - JOIN/EXISTS 建议使用 Mapper XML，自定义 SQL；
  - 或采用“两步法”：先查交集 `business_id`，再 `IN` 回表分页（注意批量大小与顺序稳定性）。

## 最佳实践
- YAML 管模型、代码管语义：YAML 定义状态/转换，枚举提供编译期约束，两者名称保持一致。
- 以 Redis 为准：任何持久化异常不影响引擎的实时读写；消费者具备幂等与补偿能力。
- 批处理优先：调大 `count` 与并发以提升吞吐，并结合 DB 批量写策略。
- 业务时间优先：确保上游事件产生的业务时间正确写入历史，消费者即可据此刷新 `updated_at`。
- 终态不可变更：默认锁定 `FINAL/ERROR`，仅白名单重开；避免“已完成被回滚”。
- 监控与告警：结合 Micrometer/OpenTelemetry 对发布、消费、落库、清理链路打点与追踪。
