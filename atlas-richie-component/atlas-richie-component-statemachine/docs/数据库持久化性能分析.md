# 数据库持久化方案性能分析

## 一、当前定时任务方案的性能瓶颈

### 1.1 核心问题

**当前实现方式：**
- 使用 `@Scheduled` 定时任务每 2 秒轮询一次
- 使用分布式锁（`optimisticLock`）确保单实例处理
- 只有获取到锁的实例才能处理，其他实例都在空转

**性能瓶颈：**

| 瓶颈点        | 说明                          | 影响            |
|------------|-----------------------------|---------------|
| **单实例处理**  | 即使有 N 个实例，每次只能有 1 个实例处理队列   | 无法利用多实例并行能力   |
| **轮询延迟**   | 固定 2 秒间隔，即使队列堆积也要等待         | 实时性差，最大延迟 2 秒 |
| **锁竞争开销**  | 多个实例频繁竞争锁，但只有一个能成功          | 浪费 CPU 和网络资源  |
| **批量大小限制** | 单次只能处理 `batchSize`（默认 200）条 | 高并发场景下可能积压    |

### 1.2 性能计算示例

**场景假设：**
- 3 个应用实例
- 每秒产生 1000 条状态变更
- 批量大小：200 条
- 每次刷写耗时：100ms

**当前方案：**
- 每 2 秒轮询一次，只有 1 个实例处理
- 处理能力：200 条/2秒 = 100 条/秒
- **积压速度：1000 - 100 = 900 条/秒**
- **积压会持续增长！**

**吞吐量瓶颈：**
```
理论最大吞吐 = batchSize / flushIntervalMs
              = 200 / 2000ms
              = 100 条/秒
```

### 1.3 适用场景

当前方案适合：
- ✅ **低并发场景**（< 100 条/秒）
- ✅ **对实时性要求不高**（容忍秒级延迟）
- ✅ **简单部署**（不想引入额外中间件）

不适合：
- ❌ **高并发场景**（> 500 条/秒）
- ❌ **对实时性要求高**（< 1 秒延迟）
- ❌ **多实例部署**（无法充分利用多实例）

## 二、Redis Stream MQ 方案分析（推荐）

### 2.1 Redis Stream MQ 架构

```
状态变更事件 → Redis Stream → 多消费者实例并行处理 → 数据库
```

**实现方式：**
- 状态变更时，发送消息到 Redis Stream（基于现有 Redis 基础设施）
- 多个消费者实例并行消费消息（Redis Stream 消费组天然支持）
- 每个消费者处理自己的消息，无需分布式锁
- **自适应轮询优化**：有消息时立即拉取（< 100ms），无消息时长轮询（2秒）

### 2.2 Redis Stream MQ 核心优势

#### ✅ **无需额外中间件**
- 基于现有 Redis 基础设施，无需部署 Kafka/RocketMQ
- 降低系统复杂度和运维成本
- 充分利用现有技术栈

#### ✅ **自适应轮询优化**（已实现）
- **有消息时**：50ms 后立即拉取，延迟 < 100ms（**提升 20 倍**）
- **无消息时**：2 秒长轮询，节省资源
- **自动调整**：根据消息情况动态调整，无需手动配置

### 2.3 性能优势对比（Redis Stream MQ 优化后）

| 指标 | 定时任务方案 | Redis Stream MQ（优化后） | 提升 |
|------|------------|------------------------|------|
| **并行度** | 1 个实例 | **N 个实例并行** | **N 倍** |
| **实时性（有消息）** | 最大 2 秒延迟 | **< 100ms 延迟** ✅ | **提升 20 倍** |
| **实时性（无消息）** | 2 秒延迟 | 2 秒长轮询 | 相同（合理） |
| **吞吐量（单实例）** | 100 条/秒 | **500-1000 条/秒** | **5-10 倍** |
| **吞吐量（多实例）** | 100 条/秒 | **N × 500 条/秒** | **N × 5 倍** |
| **资源利用** | 其他实例空转 | **所有实例工作** | **100% 利用** |
| **积压处理** | 单实例处理，积压增长 | **多实例并行，快速消费** | **显著改善** |
| **部署复杂度** | 简单 | **简单**（基于现有 Redis） | **相同** |
| **运维成本** | 低 | **低**（无需额外中间件） | **相同** |

### 2.4 Redis Stream MQ 方案优点

#### ✅ 1. **高吞吐量**
```
理论最大吞吐 = 消费者实例数 × 单实例处理能力
              = 3 × 500-1000 条/秒
              = 1500-3000 条/秒
```
- ✅ 可以水平扩展消费者实例
- ✅ 天然支持批量消费
- ✅ 自适应轮询：消息密集时可连续拉取

#### ✅ 2. **低延迟**（已优化）
- ✅ **有消息时**：50ms 后立即拉取，延迟 < 100ms
- ✅ **无消息时**：2 秒长轮询，节省资源
- ✅ **自适应策略**：根据消息情况动态调整

#### ✅ 3. **高可用性**
- ✅ Redis Stream 提供消息持久化
- ✅ 支持消息 ACK 确认机制
- ✅ 支持消息重试和错误处理策略
- ✅ 消费者宕机不影响消息持久化

#### ✅ 4. **解耦**
- ✅ 生产者与消费者完全解耦
- ✅ 可以独立扩展和部署
- ✅ 支持多消费者组并行处理

#### ✅ 5. **监控和运维**
- ✅ 集成 Micrometer 监控指标
- ✅ 支持 OpenTelemetry 链路追踪
- ✅ 可以查看队列积压情况
- ✅ 支持消息追溯

#### ✅ 6. **无需额外中间件**（核心优势）
- ✅ 基于现有 Redis 基础设施
- ✅ 无需部署 Kafka/RocketMQ
- ✅ 降低运维成本和系统复杂度

### 2.5 Redis Stream MQ 方案缺点

#### ⚠️ 1. **依赖 Redis**
- 需要 Redis 服务可用（但通常已有 Redis 基础设施）
- Redis 故障会影响持久化功能（需要保障 Redis 高可用）

#### ⚠️ 2. **吞吐量上限**
- 相比 Kafka/RocketMQ，吞吐量上限较低（500-2000 条/秒）
- 但对于大多数业务场景已足够

#### ⚠️ 3. **无消息时延迟**
- 无消息时仍为 2 秒长轮询（这是合理的，节省资源）
- 有消息时延迟已优化到 < 100ms

#### ✅ **已解决的问题**
- ✅ **系统复杂度**：无需额外中间件，基于现有 Redis
- ✅ **开发成本**：已有成熟的 Redis Stream 组件，开箱即用
- ✅ **消息顺序性**：支持幂等性，已考虑消息乱序场景

## 三、方案选择建议

### 3.1 决策矩阵（Redis Stream MQ 优化后）

| 评估维度 | 定时任务方案 | Redis Stream MQ（优化后） | 推荐 |
|---------|------------|------------------------|------|
| **并发量** | < 100/秒 | > 500/秒 | **Redis Stream MQ** |
| **实时性要求** | 秒级可接受 | **毫秒级（< 100ms）** | **Redis Stream MQ** |
| **部署复杂度** | 简单 | **简单**（基于现有 Redis） | **相同** |
| **运维成本** | 低 | **低**（无需额外中间件） | **相同** |
| **扩展性** | 差 | **优秀**（多实例并行） | **Redis Stream MQ** |
| **可靠性** | 中等 | **高**（消息持久化+ACK） | **Redis Stream MQ** |
| **多实例利用** | 差（单实例） | **优秀**（多实例并行） | **Redis Stream MQ** |

### 3.2 推荐方案（Redis Stream MQ 优化后）

#### 场景 1：低并发场景（< 200 条/秒）
**推荐：Redis Stream MQ 方案** ✅
- ✅ 简单可靠（基于现有 Redis）
- ✅ 无需额外中间件
- ✅ 性能优于定时任务（延迟更低，多实例并行）
- ✅ 为未来扩展预留空间

#### 场景 2：中高并发场景（200-2000 条/秒）
**推荐：Redis Stream MQ 方案** ✅ **强烈推荐**
- ✅ **多实例并行处理**，吞吐量提升 N 倍
- ✅ **低延迟**：有消息时 < 100ms
- ✅ **无需额外中间件**，基于现有 Redis
- ✅ **可靠性高**：消息持久化 + ACK 机制

#### 场景 3：超高并发场景（> 2000 条/秒）
**推荐：Redis Stream MQ 方案** ✅ **或考虑专业 MQ**
- ✅ Redis Stream MQ：对于大多数场景已足够（可扩展到 3000+ 条/秒）
- ⚠️ 如需更高吞吐量（> 5000 条/秒），可考虑 Kafka/RocketMQ
- ⚠️ 但需要评估额外的运维成本和复杂度

### 3.3 迁移策略（Redis Stream MQ）

**推荐：直接切换** ✅

由于 Redis Stream MQ 基于现有 Redis 基础设施，迁移成本低，建议：

1. **阶段 1：实现 Redis Stream MQ 方案**
   - 实现状态变更事件发布到 Redis Stream
   - 实现消费者处理消息并写入数据库
   - 保留定时任务代码作为降级方案

2. **阶段 2：灰度切换**
   - 通过配置开关控制是否启用 Redis Stream MQ
   - 逐步将流量从定时任务切换到 Redis Stream MQ
   - 监控性能指标和错误率

3. **阶段 3：完全切换**
   - 关闭定时任务
   - 完全依赖 Redis Stream MQ 处理
   - 保留定时任务代码作为降级方案（可选）

**优势：**
- ✅ 无需额外中间件，迁移成本低
- ✅ 基于现有 Redis，风险可控
- ✅ 可以随时回退到定时任务方案

## 四、Redis Stream MQ 方案实现建议

### 4.1 推荐使用 Redis Stream MQ（已实现）

**Redis Stream MQ 优势：**
- ✅ **无需额外中间件**：基于现有 Redis 基础设施
- ✅ **自适应轮询**：有消息时延迟 < 100ms，已优化
- ✅ **多实例并行**：天然支持多消费者并行处理
- ✅ **可靠性高**：消息持久化 + ACK 机制
- ✅ **运维简单**：无需额外部署和维护

### 4.2 实现要点

1. **消息体设计**
   ```java
   public record StateSyncMessage(String syncKey) implements BaseStreamMessage {
       // syncKey 格式：stateMachineName:businessId
   }
   ```
   - ✅ 只包含同步键，不包含状态数据
   - ✅ 保持与当前方案一致（从 Redis 读取状态）
   - ✅ 实现 `BaseStreamMessage` 接口

2. **发布消息**
   ```java
   @EventListener
   public void onStateChanged(StateChangedEvent event) {
       String syncKey = StateSyncKey.build(
           event.getStateMachineName(),
           event.getBusinessId()
       );
       // 发布到 Redis Stream
       GlobalCache.stream().publish("statemachine:db:sync", 
           new StateSyncMessage(syncKey));
   }
   ```

3. **消费者实现**
   ```java
   @RedisStreamConsumer("statemachine-db-sync")
   public class StateMachineDbSyncConsumer 
       extends AbstractStreamConsumer<StateSyncMessage> {
       
       @Autowired
       private JdbcTemplate jdbcTemplate;
       
       @Autowired
       private StateStorage stateStorage;
       
       @Override
       protected void handle(StateSyncMessage payload, EventContext ctx) throws Exception {
           StateSyncKey key = StateSyncKey.parse(payload.syncKey());
           
           // 从 Redis 读取最新状态
           String currentState = stateStorage.getCurrentState(
               key.getStateMachineName(),
               key.getBusinessId()
           );
           
           // 批量写入数据库
           // ... 批量写入逻辑
       }
   }
   ```

4. **消费者配置**
   ```yaml
   platform:
     cache:
       redis:
         stream:
           consumers:
             configs:
               statemachine-db-sync:
                 streamKey: statemachine:db:sync
                 group: db-sync-group
                 consumer: db-sync-consumer
                 concurrency: 10  # 根据数据库连接池调整
                 autoAck: true
                 errorStrategy: RETRY
   ```

5. **降级方案**
   - ✅ Redis Stream 故障时，可回退到定时任务方案
   - ✅ 通过配置开关控制是否启用 Redis Stream MQ
   - ✅ 保留定时任务代码作为降级方案

## 五、总结

### 5.1 性能对比总结（Redis Stream MQ 优化后）

| 方案 | 吞吐量（单实例） | 吞吐量（多实例） | 延迟（有消息） | 延迟（无消息） | 复杂度 | 成本 | 推荐场景 |
|------|----------------|----------------|--------------|--------------|--------|------|----------|
| **定时任务** | 100 条/秒 | 100 条/秒 | 2 秒 | 2 秒 | 低 | 低 | ❌ 不推荐 |
| **Redis Stream MQ（优化后）** | 500-1000 条/秒 | **N × 500 条/秒** | **< 100ms** ✅ | 2 秒 | **低** ✅ | **低** ✅ | ✅ **强烈推荐** |

### 5.2 最终建议（Redis Stream MQ 优化后）

**✅ 强烈推荐：使用 Redis Stream MQ 方案**

**理由：**
1. ✅ **性能显著提升**
   - 吞吐量提升 5-10 倍（单实例）
   - 多实例并行，吞吐量提升 N 倍
   - 延迟降低 20 倍（有消息时 < 100ms）

2. ✅ **无需额外中间件**
   - 基于现有 Redis 基础设施
   - 部署复杂度与定时任务相同
   - 运维成本与定时任务相同

3. ✅ **可靠性更高**
   - 消息持久化 + ACK 机制
   - 支持多实例并行处理
   - 支持错误重试和死信队列

4. ✅ **未来扩展性**
   - 可以水平扩展消费者实例
   - 自适应轮询已优化，性能优秀
   - 为未来业务增长预留空间

**迁移建议：**
- ✅ **直接采用 Redis Stream MQ 方案**，替代定时任务
- ✅ 保留定时任务代码作为降级方案（可选）
- ✅ 通过配置开关控制，支持灰度切换

**最佳实践：**
- ✅ 使用 Redis Stream MQ 作为主要方案
- ✅ 根据实际业务场景调整并发数和批量大小
- ✅ 监控消息积压、处理延迟等指标
