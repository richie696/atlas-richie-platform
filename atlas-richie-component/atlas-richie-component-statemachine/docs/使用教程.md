## 🎯 什么是状态机？

### 生活中的状态机

想象一下自动售货机的工作流程：

```
投币 → 选择商品 → 出货 → 找零
```

每个步骤都是一个**状态**，从一个状态到另一个状态的过程就是**状态转换**。

### 技术中的状态机

状态机是一种数学模型，用来描述对象在不同状态之间的转换规则。

**简单理解**：状态机就像一个"交通警察"，它规定了：
- 当前在哪个状态
- 可以转换到哪些状态
- 什么条件下可以转换
- 转换时执行什么动作

## 🤔 为什么需要状态机？

### 传统方式的问题

```java
// 传统方式：直接修改状态
public void confirmOrder(Order order) {
    if ("PENDING".equals(order.getStatus())) {
        order.setStatus("CONFIRMED");
        orderRepository.save(order);
    } else {
        throw new RuntimeException("订单状态不正确");
    }
}

public void startPrepare(Order order) {
    if ("CONFIRMED".equals(order.getStatus())) {
        order.setStatus("PREPARING");
        orderRepository.save(order);
    } else {
        throw new RuntimeException("订单状态不正确");
    }
}
```

**问题**：
- ❌ 状态转换逻辑分散在各个方法中
- ❌ 容易遗漏状态检查
- ❌ 难以维护和扩展
- ❌ 状态转换规则不清晰

### 状态机方式的优势

```java
// 状态机方式：统一管理状态转换（使用静态外观类）

import com.richie.component.statemachine.StateMachine;

public void confirmOrder(String orderId) {
    // 状态机会自动检查转换是否合法
    // 使用枚举类型，提供编译期约束
    StateTransitionResult result = StateMachine.fire(OrderSm.order, OrderEvent.CONFIRM, orderId);

    if (result.isSuccess()) {
        log.info("订单确认成功: {}", orderId);
    } else {
        throw new RuntimeException("订单确认失败: " + result.getErrorMessage());
    }
}
```

**优势**：
- ✅ 状态转换规则集中管理
- ✅ 自动验证转换合法性
- ✅ 支持条件判断和动作执行（SpEL表达式）
- ✅ 完整的状态历史记录（Redis存储，默认保留7天）
- ✅ 易于维护和扩展
- ✅ 解耦业务对象，仅需传入业务ID
- ✅ Redis作为单数据源，数据库异步持久化

## 📚 状态机核心概念

### 1. 状态（State）

**定义**：对象在某一时刻所处的条件或情况。

**示例**：
```
订单状态：待确认 → 已确认 → 制作中 → 待配送 → 配送中 → 已完成
支付状态：待支付 → 支付中 → 支付成功/支付失败
```

### 2. 事件（Event）

**定义**：触发状态转换的动作或条件。

**示例**：
```
CONFIRM（确认订单）
START_PREPARE（开始制作）
COMPLETE（完成订单）
CANCEL（取消订单）
```

### 3. 转换（Transition）

**定义**：从一个状态到另一个状态的过程。

**组成**：
- 源状态（fromState）
- 目标状态（toState）
- 触发事件（event）
- 转换条件（condition）
- 转换动作（action）

### 4. 状态机（State Machine）

**定义**：包含所有状态和转换规则的完整定义。

## ⚙️ 状态机工作原理

### 状态转换流程图

```
┌─────────────┐    CONFIRM     ┌─────────────┐    START_PREPARE    ┌─────────────┐
│   待确认    │ ──────────────→ │   已确认    │ ──────────────────→ │   制作中    │
│  (PENDING)  │                 │ (CONFIRMED) │                     │ (PREPARING) │
└─────────────┘                 └─────────────┘                     └─────────────┘
       │                               │                                   │
       │ CANCEL                         │ CANCEL                            │ READY
       ▼                               ▼                                   ▼
┌─────────────┐                 ┌─────────────┐                     ┌─────────────┐
│   已取消    │                 │   已取消    │                     │   待配送    │
│ (CANCELLED) │                 │ (CANCELLED) │                     │   (READY)   │
└─────────────┘                 └─────────────┘                     └─────────────┘
                                                                           │
                                                                           │ START_DELIVER
                                                                           ▼
                                                                   ┌─────────────┐
                                                                   │   配送中    │
                                                                   │(DELIVERING) │
                                                                   └─────────────┘
                                                                           │
                                                                           │ COMPLETE
                                                                           ▼
                                                                   ┌─────────────┐
                                                                   │   已完成    │
                                                                   │ (COMPLETED) │
                                                                   └─────────────┘
```

### 状态机执行过程

1. **接收请求**：业务代码调用静态外观类 `StateMachine.fire()`
2. **获取当前状态**：从Redis存储中获取当前状态（如果Redis为空且启用DB持久化，则从数据库加载并回填Redis）
3. **查找转换规则**：根据当前状态和事件查找匹配的转换
4. **终态检查**：如果当前状态是FINAL或ERROR，仅允许带有`attributes.reopen=true`的转换
5. **验证条件**：使用SpEL表达式检查转换条件是否满足（支持安全检查）
6. **执行转换**：更新状态并执行转换动作（SpEL表达式）
7. **保存结果**：将新状态保存到Redis（单数据源），并投递到Redis Stream用于异步持久化
8. **返回结果**：返回转换成功或失败的结果

## 🎯 何时使用状态机？

### 适用场景

#### ✅ 推荐使用状态机的场景

1. **订单管理**
   - 订单状态：待确认 → 已确认 → 制作中 → 配送中 → 已完成
   - 需要严格控制状态转换顺序

2. **支付流程**
   - 支付状态：待支付 → 支付中 → 支付成功/支付失败
   - 涉及资金安全，需要严格验证

3. **库存管理**
   - 库存状态：正常 → 预警 → 缺货 → 补货中 → 正常
   - 需要根据库存量自动调整状态

4. **审批流程**
   - 审批状态：待审批 → 审批中 → 已通过/已拒绝
   - 需要记录审批历史

5. **任务管理**
   - 任务状态：待分配 → 进行中 → 已完成/已取消
   - 需要跟踪任务进度

#### ❌ 不推荐使用状态机的场景

1. **简单的开关状态**：启用/禁用
2. **无转换规则的状态**：如用户性别、商品分类
3. **频繁变化的状态**：如实时位置、温度等

### 判断标准

**使用状态机，当你的业务满足以下条件时**：

- 🔄 有明确的状态转换规则
- 📋 需要严格控制状态转换顺序
- 📝 需要记录状态变更历史
- ⚠️ 需要防止非法状态转换
- 🎯 需要根据状态执行不同业务逻辑

## 🚀 快速开始

### 第一步：添加依赖

```xml
<dependency>
    <groupId>com.richie.component</groupId>
    <artifactId>atlas-richie-component-statemachine</artifactId>
    <version>4.5.0-SNAPSHOT</version>
</dependency>
```

### 第二步：配置状态机

```yaml
# application.yml
platform:
  component:
    statemachine:
      enabled: true
      # 规则引擎配置
      rules-engine:
        skip-on-first-failed-rule: false
        skip-on-first-applied-rule: false
        skip-on-first-non-triggered-rule: false
        priority-based: true
        rule-priority-threshold: 0
        enable-execution-log: false
        execution-timeout-ms: 0
        # 表达式配置（SpEL）
        expression:
          slow-threshold-ms: 50
          enable-security-check: true
          security-blacklist: ""
          enable-detailed-log: false
      # 状态机配置
      state-machine:
        enable-history: true
        enable-events: true
        config-path: classpath:statemachine/
      # 存储配置
      storage:
        key-prefix: platform:statemachine
        timeout: 604800000  # 历史记录过期时间（毫秒），默认7天
        # 数据库异步持久化配置
        db:
          enabled: true
          batch-size: 200
          flush-interval-ms: 2000
          buffer-capacity: 10000
```

### 第三步：创建状态机配置文件

```yaml
# src/main/resources/statemachine/order-statemachine.yml
name: order
description: 订单状态机

# 状态定义
states:
  - name: PENDING
    description: 待确认
    type: INITIAL
    
  - name: CONFIRMED
    description: 已确认
    type: NORMAL
    
  - name: COMPLETED
    description: 已完成
    type: FINAL

# 转换定义
transitions:
  - name: confirm
    description: 确认订单
    fromState: PENDING
    toState: CONFIRMED
    event: CONFIRM
    # SpEL条件表达式：可通过context访问上下文属性
    condition: "context.attributes['amount'] != null and context.attributes['amount'] > 0"
    # SpEL动作表达式：可执行业务逻辑
    action: "context.attributes['confirmedTime'] = '2025-01-01T00:00:00'"
    
  - name: complete
    description: 完成订单
    fromState: CONFIRMED
    toState: COMPLETED
    event: COMPLETE
    action: "context.attributes['completedTime'] = '2025-01-01T00:00:00'"
```

### 第四步：定义枚举类（必须与YAML名称一致）

```java
// 状态机名称枚举
public enum OrderSm {
    order
}

// 事件枚举（必须与YAML中的event名称一致）
public enum OrderEvent {
    CONFIRM,
    COMPLETE,
    CANCEL
}

// 状态枚举（必须与YAML中的state名称一致）
public enum OrderState {
    PENDING,
    CONFIRMED,
    COMPLETED,
    CANCELLED
}
```

### 第五步：使用状态机

```java
import com.richie.component.statemachine.StateMachine;
import engine.com.richie.component.statemachine.StateTransitionResult;
import storage.com.richie.component.statemachine.StateHistory;

@Service
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;

    /**
     * 确认订单（基础用法）
     */
    public void confirmOrder(String orderId) {
        // 使用静态外观类，传入业务ID即可
        StateTransitionResult result = StateMachine.fire(OrderSm.order, OrderEvent.CONFIRM, orderId);

        if (result.isSuccess()) {
            log.info("订单确认成功: {}", orderId);
        } else {
            throw new RuntimeException("订单确认失败: " + result.getErrorMessage());
        }
    }

    /**
     * 确认订单（带上下文属性）
     */
    public void confirmOrderWithContext(String orderId, String operator, Double amount) {
        // 传入上下文属性，可在SpEL表达式中使用
        Map<String, Object> attributes = Map.of(
                "operator", operator,
                "amount", amount,
                "channel", "WEB"
        );

        StateTransitionResult result = StateMachine.fire(
                OrderSm.order,
                OrderEvent.CONFIRM,
                orderId,
                attributes
        );

        if (result.isSuccess()) {
            log.info("订单确认成功: {}", orderId);
        } else {
            throw new RuntimeException("订单确认失败: " + result.getErrorMessage());
        }
    }

    /**
     * 完成订单（显式传入当前状态，避免一次Redis读取）
     */
    public void completeOrder(String orderId) {
        // 如果业务代码已知当前状态，可显式传入以提升性能
        StateTransitionResult result = StateMachine.fire(
                OrderSm.order,
                OrderEvent.COMPLETE,
                orderId,
                OrderState.CONFIRMED  // 显式传入当前状态
        );

        if (result.isSuccess()) {
            log.info("订单完成: {}", orderId);
        } else {
            throw new RuntimeException("订单完成失败: " + result.getErrorMessage());
        }
    }

    /**
     * 获取当前状态
     */
    public OrderState getCurrentState(String orderId) {
        // 获取当前状态（枚举类型）
        return StateMachine.getCurrentState(OrderSm.order, orderId, OrderState.class);
    }

    /**
     * 检查是否可以执行某个事件
     */
    public boolean canConfirm(String orderId) {
        return StateMachine.canTransitionTo(OrderSm.order, OrderEvent.CONFIRM, orderId);
    }

    /**
     * 获取状态历史
     */
    public List<StateHistory> getOrderHistory(String orderId) {
        return StateMachine.getStateHistory(OrderSm.order, orderId);
    }
}
```

#### 业务需求
- 订单从创建到完成需要经过多个状态

### 案例二：支付系统

#### 状态机配置

```yaml
# payment-statemachine.yml
name: payment
description: 支付状态机

states:
  - name: PENDING
    description: 待支付
    type: INITIAL
    
  - name: PROCESSING
    description: 支付中
    type: NORMAL
    
  - name: SUCCESS
    description: 支付成功
    type: FINAL
    
  - name: FAILED
    description: 支付失败
    type: ERROR
    
  - name: REFUNDING
    description: 退款中
    type: NORMAL
    
  - name: REFUNDED
    description: 已退款
    type: FINAL

transitions:
  - name: start_payment
    description: 开始支付
    fromState: PENDING
    toState: PROCESSING
    event: START_PAYMENT
    condition: "context.attributes['amount'] != null and context.attributes['amount'] > 0 and context.attributes['paymentMethod'] != null"
    action: "context.attributes['processStartTime'] = '2025-01-01T00:00:00'"
    
  - name: payment_success
    description: 支付成功
    fromState: PROCESSING
    toState: SUCCESS
    event: PAYMENT_SUCCESS
    action: "context.attributes['successTime'] = '2025-01-01T00:00:00'"
    
  - name: payment_failed
    description: 支付失败
    fromState: PROCESSING
    toState: FAILED
    event: PAYMENT_FAILED
    action: "context.attributes['failedTime'] = '2025-01-01T00:00:00'"
    
  - name: start_refund
    description: 开始退款（从终态重新打开）
    fromState: SUCCESS
    toState: REFUNDING
    event: START_REFUND
    condition: "context.attributes['canRefund'] != null and context.attributes['canRefund'] == true"
    # 终态重新打开需要设置 attributes.reopen: true
    attributes:
      reopen: true
```

## 📋 最佳实践

### 1. 状态设计原则

#### ✅ 好的状态设计
```yaml
states:
  - name: PENDING      # 清晰的状态名称
    description: 待确认  # 详细的状态描述
    type: INITIAL      # 明确的状态类型
    
  - name: CONFIRMED
    description: 已确认
    type: NORMAL
    
  - name: COMPLETED
    description: 已完成
    type: FINAL
```

#### ❌ 不好的状态设计
```yaml
states:
  - name: S1           # 不清晰的状态名称
    description: 状态1  # 模糊的描述
    type: NORMAL
    
  - name: PROCESSING   # 过于宽泛的状态
    description: 处理中
    type: NORMAL
```

### 2. 转换设计原则

#### ✅ 好的转换设计
```yaml
transitions:
  - name: confirm_order
    description: 确认订单
    fromState: PENDING
    toState: CONFIRMED
    event: CONFIRM
    condition: "order.amount > 0"  # 明确的条件
    action: "order.setConfirmedTime(new java.time.LocalDateTime())"  # 具体的动作
    priority: 1
```

#### ❌ 不好的转换设计
```yaml
transitions:
  - name: change_state
    description: 改变状态
    fromState: PENDING
    toState: CONFIRMED
    event: CHANGE
    # 缺少条件和动作
```

### 3. 条件表达式设计

#### ✅ 好的条件表达式
```yaml
condition: "order.amount > 0 && order.customerId != null"
condition: "payment.canRefund() && payment.refundAmount <= payment.amount"
condition: "inventory.stockQuantity > 0"
```

#### ❌ 不好的条件表达式
```yaml
condition: "true"  # 总是为真，没有实际意义
condition: "order != null"  # 过于简单
condition: "complex.business.logic.here"  # 过于复杂
```

### 4. 动作表达式设计

#### ✅ 好的动作表达式
```yaml
action: "order.setConfirmedTime(new java.time.LocalDateTime())"
action: "payment.setTransactionId(context.getAttribute('transactionId'))"
action: "inventory.updateStock(-1)"
```

#### ❌ 不好的动作表达式
```yaml
action: "doSomething()"  # 不明确的动作
action: "complex.business.logic()"  # 过于复杂的逻辑
action: ""  # 空动作
```

### 5. 错误处理

```java
@Service
public class OrderService {
    
    public void confirmOrder(String orderId, Double amount) {
        try {
            Map<String, Object> attributes = Map.of("amount", amount);
            
            StateTransitionResult result = StateMachine.fire(
                OrderSm.order, 
                OrderEvent.CONFIRM, 
                orderId, 
                attributes
            );
            
            if (result.isSuccess()) {
                log.info("订单确认成功: {}", orderId);
            } else {
                // 记录详细的错误信息
                log.error("订单确认失败: {}, 错误: {}", orderId, result.getErrorMessage());
                throw new BusinessException("订单确认失败: " + result.getErrorMessage());
            }
        } catch (Exception e) {
            log.error("订单确认异常: {}", orderId, e);
            throw new BusinessException("订单确认异常", e);
        }
    }
}
```

```java
public class OrderService {
```

### 7. SpEL表达式使用

SpEL表达式用于条件判断和动作执行，可通过`context`访问上下文：

**可用变量**：
- `context`：完整上下文对象（StateContext）
- `#currentState`：当前状态字符串
- `#previousState`：前一状态字符串
- `#event`：当前触发的事件名
- `context.attributes['key']`：获取上下文属性
- `context.attributes['key'] = value`：设置上下文属性

**条件表达式示例**：
```yaml
condition: "context.attributes['amount'] != null and context.attributes['amount'] > 0"
condition: "context.attributes['operatorRole'] == 'ADMIN'"
condition: "context.attributes['canCancel'] == true"
```

**动作表达式示例**：
```yaml
action: "context.attributes['confirmedTime'] = '2025-01-01T00:00:00'"
action: "context.attributes['operator'] = #event"
```

### 8. 表达式与安全

状态机使用 **Spring SpEL (Spring Expression Language)** 作为表达式引擎，并采用 **沙箱模式** 提供强大的安全保障。

#### SpEL 沙箱机制

状态机使用 `SpelExpressionParser` + `SimpleEvaluationContext.forReadOnlyDataBinding().build()` 创建沙箱环境：

```java
SpelExpressionParser parser = new SpelExpressionParser();
SimpleEvaluationContext evalContext = SimpleEvaluationContext.forReadOnlyDataBinding().build();
```

**沙箱提供的安全保证**：

| 操作类型 | 沙箱行为 |
|---------|---------|
| 方法调用 | ❌ 禁止（如 `context.toString()`、`System.getProperty()`） |
| 类型引用 | ❌ 禁止（如 `T(java.lang.Runtime)`） |
| 构造调用 | ❌ 禁止（如 `new HashMap()`） |
| Bean 引用 | ❌ 禁止（如 `@beanName`） |
| 属性访问 | ✅ 允许（如 `#currentState`、`context.attributes['key']`） |

#### 深度防御：黑名单机制

除沙箱外，还保留了**黑名单机制**作为深度防御：

```yaml
platform:
  component:
    statemachine:
      expression:
        enable-security-check: true
        security-blacklist:  # 自定义黑名单
          - java.lang.Runtime
          - java.lang.ProcessBuilder
          - javax.script.ScriptEngine
```

#### 4 个被阻止的攻击向量示例

| 攻击类型 | MVEL 写法（危险） | SpEL 沙箱行为 |
|---------|------------------|--------------|
| 方法调用 | `context.toString()` | ❌ SpEL 拒绝 |
| 类型引用 | `T(java.lang.Runtime)` | ❌ SpEL 拒绝 |
| 构造调用 | `new java.util.HashMap()` | ❌ SpEL 拒绝 |
| 正常属性 | `#currentState == 'PENDING'` | ✅ 正常工作 |

#### 为什么选择 SpEL 而非 MVEL

| 维度 | MVEL | SpEL |
|------|------|------|
| **维护状态** | 已停止维护（最后更新 2017） | Spring 活跃维护 |
| **安全机制** | 仅黑名单（可绕过） | 原生沙箱（引擎级） |
| **RCE 漏洞** | Dependobot #396，无修复 | 无已知 RCE |
| **类型安全** | 弱类型 | 强类型 |
| **Spring 集成** | 无 | 原生集成 |

#### SpEL 语法速查表

| MVEL 写法 | SpEL 写法 | 说明 |
|----------|----------|------|
| `context.getAttribute('amount', 0)` | `context.attributes['amount']` | Map 访问 |
| `context.getAttribute('amount', 0) > 0` | `context.attributes['amount'] != null and context.attributes['amount'] > 0` | 条件判断 |
| `currentState == 'PENDING'` | `#currentState == 'PENDING'` | 变量访问 |
| `event == 'CONFIRM'` | `#event == 'CONFIRM'` | 事件访问 |
| `attributes['level'] == 'high'` | `attributes['level'] == 'high'` | 属性访问 |
| `java.time.LocalDateTime.now()` | ❌ 禁止 | 方法调用 |
| `Runtime.getRuntime().exec(...)` | ❌ 禁止 | 类型+方法 |
| `new java.util.HashMap()` | ❌ 禁止 | 构造调用 |
| `T(java.lang.Runtime)` | ❌ 禁止 | 类型引用 |

#### 可用变量

SpEL 表达式中可使用以下变量：

| 变量名 | 说明 | 示例 |
|-------|------|------|
| `#context` | 完整上下文对象 | `#context.attributes['amount']` |
| `#currentState` | 当前状态字符串 | `#currentState == 'PENDING'` |
| `#previousState` | 前一状态字符串 | `#previousState == 'CONFIRMED'` |
| `#event` | 当前事件名 | `#event == 'CONFIRM'` |
| `#transition` | 转换信息对象 | `#transition.name` |
| `attributes` | 注入的键值对 | `attributes['operator']` |

#### 配置项

| 配置项 | 类型 | 默认值 | 说明 |
|-------|------|-------|------|
| `expression.slow-threshold-ms` | long | 1000 | 慢查询阈值（毫秒） |
| `expression.enable-security-check` | boolean | true | 是否启用安全检查 |
| `expression.security-blacklist` | list | `[]` | 自定义表达式黑名单 |
| `expression.enable-detailed-log` | boolean | false | 是否启用详细日志 |

### 9. 终态不可变更规则

状态类型为`FINAL`或`ERROR`的状态默认不允许转换，除非转换配置中明确设置`attributes.reopen: true`：

```yaml
transitions:
  - name: reopen_payment
    description: 重新打开已完成的支付
    fromState: SUCCESS  # FINAL状态
    toState: REFUNDING
    event: START_REFUND
    attributes:
      reopen: true  # 允许从终态转换
```

## ❓ 常见问题

### Q1: 什么时候创建状态机？

**A**: 当你的业务有以下特征时，应该考虑使用状态机：
- 有明确的状态转换规则
- 需要严格控制状态转换顺序
- 需要记录状态变更历史
- 需要防止非法状态转换

### Q2: 状态机配置文件和代码的关系？

**A**: 
- **配置文件（YAML/JSON）**：定义状态机的拓扑结构（状态、转换、条件、动作）
- **枚举类（Java）**：提供编译期约束，枚举名称必须与YAML中的名称完全一致
- **业务代码**：使用静态外观类`StateMachine`触发状态转换
- **关系**：配置文件决定"能做什么"，枚举提供类型安全，代码决定"什么时候做"

### Q3: 如何处理复杂的状态转换逻辑？

**A**: 
1. **拆分为多个简单转换**：将复杂逻辑拆分为多个步骤
2. **使用条件表达式**：在转换中定义条件
3. **使用动作表达式**：在转换中执行复杂逻辑
4. **结合业务代码**：在状态转换前后执行业务逻辑

### Q4: 状态机存储和业务表存储的关系？

**A**: 
- **Redis（单数据源）**：存储当前状态和历史记录，提供高性能访问
  - 当前状态：永不过期
  - 历史记录：默认保留7天（可配置）
- **数据库（异步持久化）**：通过Redis Stream异步批量写入，用于长期存储和查询
- **业务表**：存储业务数据（订单信息、用户信息等），与状态机存储解耦
- **关系**：Redis是单数据源，数据库是异步备份，业务表与状态机存储完全解耦

### Q5: 如何调试状态机问题？

**A**: 
1. **检查配置文件**：确保状态和转换定义正确
2. **查看日志**：状态机会记录详细的转换日志
3. **验证条件**：检查转换条件是否满足
4. **测试转换**：使用单元测试验证状态转换

### Q6: 状态机性能如何？

**A**: 
- **Redis存储（当前实现）**：高性能，支持集群，作为单数据源
- **缓存预热**：启动时如果Redis为空，自动从数据库加载并回填Redis
- **异步持久化**：通过Redis Stream批量异步写入数据库，不影响主流程性能
- **历史记录过期**：默认7天自动过期，避免Redis数据量过大
- **建议**：适合高并发场景，状态查询延迟低（毫秒级）

### Q7: 为什么使用ID模式而不是传入业务对象？

**A**: 
- **解耦**：状态机组件与业务对象解耦，不依赖业务对象的字段结构
- **灵活性**：支持任意类型的业务ID（String、Long、UUID等）
- **性能**：避免反射操作，提升性能
- **简洁**：API更简洁，只需传入业务ID和上下文属性
