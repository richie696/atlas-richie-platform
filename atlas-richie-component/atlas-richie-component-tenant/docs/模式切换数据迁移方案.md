# 模式切换数据迁移方案

## 1. 迁移路径总览

`TenantModeGuard` 允许从低隔离等级向高等级迁移（`level` 递增）。以下为所有合法迁移路径及数据迁移要求：

| # | 源模式 | 目标模式 | level 变化 | 数据迁移方式 | 目标库实例 |
|---|--------|---------|-----------|------------|-----------|
| A | COLUMN | TABLE | 1→2 | 同库内按 `tenant_id` 拆分到 `_{tenant}` 后缀表 | 共享库 |
| B | COLUMN | SCHEMA | 1→3 | 跨 schema：`INSERT INTO target_schema.table SELECT * FROM shared.table WHERE tenant_id = ?` | 共享库 |
| C | COLUMN | DATABASE | 1→4 | 跨库：全表复制到独立数据库，过滤租户数据 | 独立库 |
| D | TABLE | SCHEMA | 2→3 | 跨 schema：整表复制（无需 `WHERE` 过滤，表已按租户隔离） | 共享库 |
| E | TABLE | DATABASE | 2→4 | 跨库：整表复制到独立数据库 | 独立库 |
| F | SCHEMA | DATABASE | 3→4 | 跨库：schema 内全部表复制到独立数据库 | 独立库 |

**禁止的降级路径**（`TenantModeGuard` 拒绝）：
- TABLE → COLUMN（需合并多张租户表回单表，数据冲突风险高）
- SCHEMA → COLUMN / TABLE（需跨 schema 合并，共享连接可能污染）
- DATABASE → 任何低级模式（独立库合并回共享库，数据冲突风险高）

> 降级迁移仅在极端场景（如数据库实例裁撤）下由 DBA 手动执行，不在自动化方案中。

## 2. 迁移前置条件

| 条件 | 说明 |
|------|------|
| 租户状态 | `status = ACTIVE`，非 MIGRATING / DISABLED / ARCHIVED |
| `TenantModeGuard` | `from.canMigrateTo(to)` 返回 `true` |
| 迁移锁 | 同一租户同时只能有一个迁移任务 |
| 目标资源就绪 | DATABASE 模式：目标数据库实例已由 DBA 创建；SCHEMA 模式：共享库连接具备 `CREATE SCHEMA` 权限 |
| YAML `table-templates` | TABLE 模式：已在 YAML 中声明需复制的表清单（用于自动建表） |
| 维护窗口 | 建议在低峰期执行；`status=MIGRATING` 期间该租户所有请求返回 503（`TenantErrorCode.TENANT_MIGRATING`） |

## 3. 各迁移路径详细方案

### 路径 A：COLUMN → TABLE

**场景**：某租户数据量过大导致单表查询变慢，需要从共享表拆分为独立租户表。

**迁移步骤**：

```sql
-- Step 1: 为目标表模板创建租户专用表（每个表模板执行一次）
CREATE TABLE IF NOT EXISTS t_order_tenantA LIKE t_order;
CREATE TABLE IF NOT EXISTS t_user_tenantA   LIKE t_user;

-- Step 2: 按 tenant_id 拷贝数据
INSERT INTO t_order_tenantA SELECT * FROM t_order WHERE tenant_id = 'tenantA';
INSERT INTO t_user_tenantA   SELECT * FROM t_user   WHERE tenant_id = 'tenantA';

-- Step 3: 校验
SELECT COUNT(*) FROM t_order WHERE tenant_id = 'tenantA';  -- 源表
SELECT COUNT(*) FROM t_order_tenantA;                       -- 目标表
-- 两者应完全一致
```

**验证清单**：
- [ ] 每张模板表都创建了对应的租户表
- [ ] 每张表的行数校验通过（源 WHERE tenant_id = ? vs 目标 COUNT(*)）
- [ ] 抽样 checksum 校验（`SELECT MD5(GROUP_CONCAT(col1, col2, ...)) FROM t_order_tenantA`）

**回滚**：目标模式下 `TenantModeGuard` 拒绝 TABLE→COLUMN 降级。如需回滚，保留源表数据（不删除 `WHERE tenant_id = ?` 的记录），将 `isolation_mode` 改回 COLUMN 即可——但**禁止**：`TenantModeGuard` 会在此操作时抛出异常，需人工介入。

> 建议：迁移完成后保留源数据 7 天作为回滚窗口，7 天后由 DBA 手动清理。

### 路径 B：COLUMN → SCHEMA

**场景**：中等规模 PostgreSQL 租户需要 schema 级物理隔离，但仍共享数据库实例。

**迁移步骤**（以 PostgreSQL 为例）：

```sql
-- Step 1: 创建租户 schema
CREATE SCHEMA IF NOT EXISTS tenant_tenantA;

-- Step 2: 在目标 schema 中建表（复制共享 schema 表结构）
CREATE TABLE tenant_tenantA.t_order  (LIKE public.t_order  INCLUDING ALL);
CREATE TABLE tenant_tenantA.t_user   (LIKE public.t_user   INCLUDING ALL);

-- Step 3: 拷贝租户数据
INSERT INTO tenant_tenantA.t_order SELECT * FROM public.t_order WHERE tenant_id = 'tenantA';
INSERT INTO tenant_tenantA.t_user  SELECT * FROM public.t_user  WHERE tenant_id = 'tenantA';

-- Step 4: 校验
SELECT COUNT(*) FROM public.t_order WHERE tenant_id = 'tenantA';
SELECT COUNT(*) FROM tenant_tenantA.t_order;
```

**注意事项**：
- `LIKE ... INCLUDING ALL` 复制索引、默认值、约束等，但**不复制外键**，需单独处理。
- Schema 模式下 `search_path` 默认为租户 schema，无需在 SQL 中加 schema 前缀。
- MySQL 不支持 schema 模式，此路径仅适用于 PostgreSQL / Oracle。

### 路径 C：COLUMN → DATABASE

**场景**：核心租户需要独立数据库实例，实现最强物理隔离。

**前提**：目标数据库实例已由 DBA 创建，连接信息已注册到管理 API。

**迁移步骤**：

```sql
-- Step 1: 在目标数据库建表（结构与共享库一致）
-- 通过 DBA 工具（Flyway / Liquibase）在目标库执行建表 DDL
-- 或导出共享库表结构再导入目标库

-- Step 2: 跨库拷贝（使用 dual-datasource JDBC）
-- 伪代码逻辑：
-- sourceDS = DataSourceRegistry.get("shared")
-- targetDS = DataSourceRegistry.get("tenantA")
-- for each table in tableTemplates:
--     SourceTable → copyRows(tenant_id = 'tenantA') → TargetTable

-- Step 3: 校验
-- 对每张表校验行数 + checksum
```

**Dual-Datasource 拷贝方案**（Java 伪代码）：

```java
// TenantDataMigrator 核心逻辑
public void migrateColumnToDatabase(String tenantId, DataSource targetDS) {
    DataSource sourceDS = dataSourceRegistry.get("shared");
    List<String> tables = properties.getTableTemplates(); // 或从共享库查询所有表

    for (String table : tables) {
        try (Connection src = sourceDS.getConnection();
             Connection tgt = targetDS.getConnection()) {

            // 读取源数据
            PreparedStatement ps = src.prepareStatement(
                "SELECT * FROM " + table + " WHERE tenant_id = ?");
            ps.setString(1, tenantId);
            ResultSet rs = ps.executeQuery();

            // 批量写入目标
            String insertSQL = buildInsertSQL(table, rs.getMetaData());
            PreparedStatement insert = tgt.prepareStatement(insertSQL);
            tgt.setAutoCommit(false);

            int batchSize = 0;
            while (rs.next()) {
                setParameters(insert, rs);
                insert.addBatch();
                if (++batchSize % 1000 == 0) {
                    insert.executeBatch();
                    tgt.commit();
                }
            }
            insert.executeBatch();
            tgt.commit();
        }
    }
}
```

### 路径 D：TABLE → SCHEMA

**场景**：table 模式下的租户数据量增长，需要 schema 级隔离（仅 PostgreSQL）。

**与路径 B 的区别**：源表已是租户专用表（如 `t_order_tenantA`），无需 `WHERE tenant_id = ?` 过滤。

```sql
-- Step 1: 创建目标 schema
CREATE SCHEMA IF NOT EXISTS tenant_tenantA;

-- Step 2: 建表 + 拷贝（无需 WHERE 过滤，整表复制）
CREATE TABLE tenant_tenantA.t_order  (LIKE public.t_order  INCLUDING ALL);
INSERT INTO tenant_tenantA.t_order  SELECT * FROM public.t_order_tenantA;

-- Step 3: 校验
SELECT COUNT(*) FROM public.t_order_tenantA;
SELECT COUNT(*) FROM tenant_tenantA.t_order;
```

### 路径 E：TABLE → DATABASE

**场景**：table 模式租户需要独立数据库。

与路径 C 类似但源数据无需 `WHERE tenant_id = ?` 过滤——表已经按租户隔离。

```sql
-- 目标库建表后，从源表直接全量拷贝
-- source: shared_db.t_order_tenantA  →  target: tenantA_db.t_order
```

### 路径 F：SCHEMA → DATABASE

**场景**：schema 模式租户需要独立数据库（如合规要求升级）。

**迁移步骤**：

```sql
-- PostgreSQL pg_dump / pg_restore 方案：
-- Step 1: 导出目标 schema
pg_dump -h shared_host -U app -n tenant_tenantA shared_db > tenantA_schema.sql

-- Step 2: 导入目标数据库
psql -h target_host -U app -d tenantA_db < tenantA_schema.sql

-- 或使用 JDBC 的 schema 间复制：
-- INSERT INTO tenantA_db.public.t_order SELECT * FROM shared_db.tenant_tenantA.t_order
```

> **推荐**：SCHEMA → DATABASE 迁移优先使用数据库原生工具（`pg_dump`/`pg_restore`），避免 JDBC 逐行拷贝的性能瓶颈。

## 4. 迁移安全机制

### 4.1 迁移锁定

迁移期间该租户进入 `MIGRATING` 状态，`TenancyStrategyInterceptor` 在 `beforePrepare` 阶段检测到即返回 503（`TenantErrorCode.TENANT_MIGRATING`）：

```java
// TenantStrategyInterceptor.beforePrepare()
TenantInfo info = tenantInfoProvider.getTenantInfo(tenantId);
if (info.getStatus() == TenantStatus.MIGRATING) {
    throw new TenantMigratingException(
        TenantErrorCode.TENANT_MIGRATING, tenantId);
}
```

### 4.2 数据校验

每个迁移步骤需通过行数校验，迁移完成后全量 checksum 校验：

| 校验项 | 方法 | 失败处理 |
|--------|------|---------|
| 行数 | `SELECT COUNT(*)` 源 vs 目标 | 立即中止，触发回滚 |
| Checksum | `MD5(GROUP_CONCAT(...))` 或 `SELECT * FOR XML RAW` | 标记 `VERIFY_FAILED`，人工介入 |
| 索引 | 检查目标表索引是否与源一致 | 自动重建缺失索引 |

### 4.3 迁移日志

所有迁移操作写入 `sys_tenant_migration_log`：

```sql
CREATE TABLE IF NOT EXISTS sys_tenant_migration_log (
    id              BIGINT PRIMARY KEY,
    tenant_id       VARCHAR(64)   NOT NULL,
    source_mode     VARCHAR(20)   NOT NULL,  -- COLUMN / TABLE / SCHEMA / DATABASE
    target_mode     VARCHAR(20)   NOT NULL,
    status          VARCHAR(20)   NOT NULL,  -- PENDING / IN_PROGRESS / VERIFYING / COMPLETED / FAILED / ROLLED_BACK
    started_at      DATETIME,
    completed_at    DATETIME,
    row_counts_json TEXT,                    -- {"t_order": [1000, 1000], "t_user": [500, 500]}
    error_message   TEXT,
    created_by      VARCHAR(64),
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

> 所有迁移日志受 `sys_tenant_change_log` 审计（`action_type = 'MODE_MIGRATE'`），二次不可篡改。

## 5. 迁移工具设计

### 5.1 TenantDataMigrator 接口

```java
/**
 * 租户数据迁移器 SPI。
 * 不同迁移路径有不同实现，通过 sourceMode + targetMode 组合定位。
 */
public interface TenantDataMigrator {

    /** 是否支持指定的迁移路径 */
    boolean supports(IsolationMode source, IsolationMode target);

    /**
     * 执行数据迁移。
     * @return MigrationResult 包含行数校验结果
     */
    MigrationResult migrate(MigrationContext context);

    /** 回滚迁移（在源数据未删除的前提下） */
    void rollback(MigrationContext context);

    /** 校验迁移结果（行数 + checksum） */
    VerificationResult verify(MigrationContext context);
}
```

**内置实现**：

| 实现类 | 覆盖路径 |
|--------|---------|
| `ColumnToTableMigrator` | COLUMN → TABLE |
| `ColumnToSchemaMigrator` | COLUMN → SCHEMA |
| `ToDatabaseMigrator` | COLUMN/TABLE/SCHEMA → DATABASE |
| `TableToSchemaMigrator` | TABLE → SCHEMA |

### 5.2 迁移任务状态机

```
┌─────────┐    发起    ┌─────────────┐   开始拷贝   ┌──────────────┐
│ PENDING │ ────────→ │ IN_PROGRESS │ ──────────→ │  VERIFYING   │
└─────────┘           └─────────────┘             └──────┬───────┘
                             │                           │
                             │ 拷贝失败                   │ 校验通过
                             ▼                           ▼
                      ┌──────────┐               ┌───────────┐
                      │  FAILED  │               │ COMPLETED │
                      └──────────┘               └───────────┘
                                                        │
                                                 手动回滚 ▼
                                               ┌────────────┐
                                               │ ROLLED_BACK│
                                               └────────────┘
```

### 5.3 管理 API

```
POST   /api/tenants/{tenantId}/migrate       # 发起迁移
GET    /api/tenants/{tenantId}/migrate/status # 查询迁移进度
POST   /api/tenants/{tenantId}/migrate/rollback # 手动回滚
```

## 6. 回滚策略

回滚仅在源数据**未被删除**的前提下可行。分三种情况：

| 场景 | 回滚方式 | 恢复时间 |
|------|---------|---------|
| 迁移刚完成，源数据未删 | 将 `isolation_mode` 改回源模式，切换到源数据源 | < 1 分钟（改一行记录） |
| 源数据已删 | 从备份恢复源数据 → 改回 `isolation_mode` | 数小时（依赖备份恢复速度） |
| 目标数据已写入新业务数据 | 需合并回源 → **禁止自动回滚，人工介入** | 不确定 |

**安全实践**：
- 迁移完成后，源数据**至少保留 7 天**再清理
- 回滚窗口内，通过监控确认目标模式运行正常（无异常、无性能退化）
- 回滚确认后，由 DBA 手动清理源数据

## 7. 人工迁移 SOP 模板

当自动迁移不可用时（如初始阶段 `TenantDataMigrator` 为占位实现），按以下 SOP 执行：

```
1. 【通知】提前 48h 通知租户，计划维护窗口（建议凌晨 2:00-4:00）

2. 【锁定】管理 API 将租户状态置为 MIGRATING
   POST /api/tenants/{tenantId}/status { "status": "MIGRATING" }

3. 【备份】对源数据库执行全量备份
   mysqldump / pg_dump → S3/OSS

4. 【迁移】按 §3 中对应路径的 SQL 模板执行数据迁移

5. 【校验】行数 + checksum 校验通过

6. 【切换】更新 sys_tenant:
   isolation_mode = <目标模式>
   status = ACTIVE
   -- 注：TenantModeGuard.validate() 在此步被绕过，因为执行人是 DBA

7. 【验证】通过管理 API 健康检查端点验证新数据源可用
   GET /api/tenants/{tenantId}/health → 200 OK

8. 【监控】观察 1 小时，确认无异常

9. 【清理】7 天后删除源数据
```

> 所有人工步骤必须在 `sys_tenant_change_log` 中留下 `MODE_MIGRATE` 记录，含操作人+时间+旧/新模式。
