## 持久层路由与拦截器集成模块详细设计

### 1. 模块概述

持久层路由与拦截器集成模块是多租户隔离的核心执行层，负责将租户上下文和策略转换为实际的数据源路由、SQL 改写、租户字段填充，并在事务内提供租户冻结、切换检测等安全保障。模块由四个核心部分组成：

- **动态数据源路由**：基于 `AbstractRoutingDataSource` 实现 Database 模式的数据源切换。
- **MyBatis-Plus 拦截器链**：按固定顺序组装列隔离、表名替换、攻击防护等拦截器，与策略模块联动。
- **事务管理增强**：通过 `TransactionSynchronization` 冻结租户，AOP 检测非法切换并回滚。
- **自动填充与连接清理**：`MetaObjectHandler` 填充租户字段，连接归还前重置 Schema 状态。

### 2. 动态数据源路由

#### 2.1 DynamicTenantDataSource

```java
public class DynamicTenantDataSource extends AbstractRoutingDataSource {

    private final Map<Object, DataSource> tenantDataSources = new ConcurrentHashMap<>();
    private final Object lock = new Object();
    private DataSource sharedDataSource;

    public DynamicTenantDataSource(DataSource sharedDataSource) {
        this.sharedDataSource = sharedDataSource;
        // 默认目标数据源为共享库
        setDefaultTargetDataSource(sharedDataSource);
        afterPropertiesSet();
    }

    @Override
    protected Object determineCurrentLookupKey() {
        String key = DataSourceContextHolder.get();
        return key != null ? key : "shared";
    }

    /**
     * 动态添加租户数据源（运行时）
     */
    public void addTenantDataSource(String tenantId, DataSource dataSource) {
        synchronized (lock) {
            tenantDataSources.put(tenantId, dataSource);
            Map<Object, Object> targetMap = new HashMap<>(tenantDataSources);
            targetMap.put("shared", sharedDataSource);
            setTargetDataSources(targetMap);
            afterPropertiesSet(); // 刷新解析缓存
        }
    }

    /**
     * 删除租户数据源（谨慎操作）
     */
    public void removeTenantDataSource(String tenantId) {
        synchronized (lock) {
            tenantDataSources.remove(tenantId);
            Map<Object, Object> targetMap = new HashMap<>(tenantDataSources);
            targetMap.put("shared", sharedDataSource);
            setTargetDataSources(targetMap);
            afterPropertiesSet();
        }
    }
}
```

**DataSourceContextHolder**（`TransmittableThreadLocal` 实现，与策略模块中一致）：

```java
public class DataSourceContextHolder {
    private static final TransmittableThreadLocal<String> HOLDER = new TransmittableThreadLocal<>();
    public static void set(String key) { HOLDER.set(key); }
    public static String get() { return HOLDER.get(); }
    public static void clear() { HOLDER.remove(); }
```

#### 2.2 数据源初始化

自动配置阶段根据 YAML 构建 `DynamicTenantDataSource` 并注册所有租户数据源：

```java
@Configuration
public class MultiTenancyAutoConfiguration {

    @Bean
    public DataSource dataSource(MultiTenancyProperties properties) {
        // 构建共享数据源
        MultiTenancyProperties.SharedDataSourceConfig sharedCfg = properties.getDatasource().getShared();
        HikariDataSource sharedDs = createHikariDataSource(sharedCfg, null);

        DynamicTenantDataSource dynamicDs = new DynamicTenantDataSource(sharedDs);

        // 构建租户数据源，若租户未显式配置 Hikari 参数则继承共享配置
        Map<String, MultiTenancyProperties.TenantDataSourceConfig> tenantsCfg = properties.getDatasource().getTenants();
        if (tenantsCfg != null) {
            tenantsCfg.forEach((tenantId, tenantCfg) -> {
                HikariDataSource tenantDs = createHikariDataSource(tenantCfg, sharedCfg);
                dynamicDs.addTenantDataSource(tenantId, tenantDs);
            });
        }
        return dynamicDs;
    }

    /**
     * 根据配置创建 HikariDataSource，支持 tenant 配置回退到 shared 配置。
     * @param primary 主要配置（共享或租户）
     * @param fallback 回退配置（可为 null，当 primary 为共享时不需要回退）
     */
    private HikariDataSource createHikariDataSource(
            Object primary, MultiTenancyProperties.SharedDataSourceConfig fallback) {

        HikariConfig hikariConfig = new HikariConfig();

        if (primary instanceof MultiTenancyProperties.SharedDataSourceConfig shared) {
            hikariConfig.setJdbcUrl(shared.getUrl());
            hikariConfig.setUsername(shared.getUsername());
            hikariConfig.setPassword(shared.getPassword());
            applyHikariProperties(shared.getHikari(), hikariConfig);
        } else if (primary instanceof MultiTenancyProperties.TenantDataSourceConfig tenant) {
            hikariConfig.setJdbcUrl(tenant.getUrl());
            hikariConfig.setUsername(tenant.getUsername());
            hikariConfig.setPassword(tenant.getPassword());
            // 若租户提供了自己的 Hikari 配置则使用，否则使用共享配置作为默认
            if (tenant.getHikari() != null) {
                applyHikariProperties(tenant.getHikari(), hikariConfig);
            } else if (fallback != null) {
                applyHikariProperties(fallback.getHikari(), hikariConfig);
            }
        }
        return new HikariDataSource(hikariConfig);
    }

    private void applyHikariProperties(HikariConfig source, HikariConfig target) {
        // 将 source 中的非默认值复制到 target，避免覆盖已设置的 url 等
        if (source.getMaximumPoolSize() > 0) target.setMaximumPoolSize(source.getMaximumPoolSize());
        if (source.getMinimumIdle() > 0) target.setMinimumIdle(source.getMinimumIdle());
        if (source.getIdleTimeout() > 0) target.setIdleTimeout(source.getIdleTimeout());
        if (source.getConnectionTimeout() > 0) target.setConnectionTimeout(source.getConnectionTimeout());
        // 可按需复制其他属性
    }
}
```

#### 2.3 线程安全保证

- 新增/删除数据源使用 `synchronized` 保护，并通过 `afterPropertiesSet()` 刷新内部解析缓存，该方法是线程安全的。
- `determineCurrentLookupKey()` 仅读取线程级上下文，无竞争。
- 运行时不鼓励频繁增删数据源，若需动态管理，应结合健康检查和连接池优雅关闭。

### 3. MyBatis-Plus 拦截器链

#### 3.1 拦截器顺序与职责

| 顺序 | 拦截器                             | 职责                         | 适用模式        |
| ---- | ---------------------------------- | ---------------------------- | --------------- |
| 1    | `BlockAttackInnerInterceptor`      | 阻止全表更新/删除            | 所有模式        |
| 2    | `DynamicTableNameInnerInterceptor` | 表名替换                     | Table / Hybrid  |
| 3    | `TenantLineInnerInterceptor`       | 列隔离条件注入               | Column / Hybrid |
| 4    | `PaginationInnerInterceptor`       | 分页处理（需在租户条件之后） | 所有模式        |

> 说明：`DynamicTableNameInnerInterceptor` 和 `TenantLineInnerInterceptor` 根据当前隔离模式动态启用，由 `TenancyStrategy` 决定是否生效。Hybrid 模式下两者可能同时存在，但具体租户只会触发对应策略。

#### 3.2 MybatisPlusInterceptor 配置

```java
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor(
        MultiTenancyProperties properties,
        TenantInfoProvider tenantInfoProvider) {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

    // 1. 攻击防护
    interceptor.addInnerInterceptor(new BlockAttackInnerInterceptor());

    // 2. 动态表名（Table 模式）
    DynamicTableNameInnerInterceptor tableInterceptor = new DynamicTableNameInnerInterceptor();
    tableInterceptor.setTableNameHandler((sql, tableName) -> {
        if (properties.getIgnoreTables().contains(tableName)) {
            return tableName;
        }
        String suffix = TableSuffixHolder.get();
        return suffix != null ? tableName + suffix : tableName;
    });
    interceptor.addInnerInterceptor(tableInterceptor);

    // 3. 租户字段注入（Column 模式）
    TenantLineInnerInterceptor tenantInterceptor = new TenantLineInnerInterceptor(
            properties, tenantInfoProvider);
    interceptor.addInnerInterceptor(tenantInterceptor);

    // 4. 分页
    PaginationInnerInterceptor paginationInterceptor = new PaginationInnerInterceptor();
    paginationInterceptor.setDbType(DbType.POSTGRE_SQL); // 可从配置获取
    interceptor.addInnerInterceptor(paginationInterceptor);

    return interceptor;
}
```

#### 3.3 TenantLineInnerInterceptor 实现

该拦截器继承 MyBatis-Plus 提供的基类 `JsqlParserSupport`，对 SQL 进行解析并追加租户条件。

```java
public class TenantLineInnerInterceptor extends JsqlParserSupport implements InnerInterceptor {

    private final MultiTenancyProperties properties;
    private final TenantInfoProvider tenantInfoProvider;

    public TenantLineInnerInterceptor(MultiTenancyProperties properties,
                                      TenantInfoProvider tenantInfoProvider) {
        this.properties = properties;
        this.tenantInfoProvider = tenantInfoProvider;
    }

    @Override
    public void beforeQuery(Executor executor, MappedStatement ms, Object parameter,
                            RowBounds rowBounds, ResultHandler resultHandler, BoundSql boundSql) {
        // 由父类 JsqlParserSupport 自动调用 processSelect
    }

    @Override
    public void beforePrepare(StatementHandler sh, Connection connection, Integer transactionTimeout) {
        // 由父类 JsqlParserSupport 自动调用 processUpdate/processDelete
    }

    @Override
    protected void processSelect(Select select, int index, String sql, Object obj) {
        if (!shouldProcess()) return;
        PlainSelect plainSelect = (PlainSelect) select.getSelectBody();
        if (plainSelect.getFromItem() instanceof Table) {
            String tableName = ((Table) plainSelect.getFromItem()).getName();
            if (properties.getIgnoreTables().contains(tableName)) return;
        }
        addTenantCondition(plainSelect.getWhere(), plainSelect);
    }

    @Override
    protected void processUpdate(Update update, int index, String sql, Object obj) {
        if (!shouldProcess()) return;
        if (update.getTable() != null && properties.getIgnoreTables().contains(update.getTable().getName())) return;
        addTenantCondition(update.getWhere(), update);
    }

    @Override
    protected void processDelete(Delete delete, int index, String sql, Object obj) {
        if (!shouldProcess()) return;
        if (delete.getTable() != null && properties.getIgnoreTables().contains(delete.getTable().getName())) return;
        addTenantCondition(delete.getWhere(), delete);
    }

    @Override
    protected void processInsert(Insert insert, int index, String sql, Object obj) {
        if (!shouldProcess()) return;
        if (insert.getTable() != null
            && properties.getIgnoreTables().contains(insert.getTable().getName())) return;

        Long tenantId = TenantContext.getTenantId();  // Long 类型
        String tenantCol = properties.getTenantIdColumn();

        // 如果 column 列表中已有 tenant_id，跳过（避免与 MetaObjectHandler 重复填充）
        if (insert.getColumns() != null) {
            for (Column col : insert.getColumns()) {
                if (col.getColumnName().equals(tenantCol)) return;
            }
        }

        // 追加 tenant_id 到 column 列表
        Column tenantColumn = new Column(tenantCol);
        if (insert.getColumns() == null) {
            insert.setColumns(new ColumnList(List.of(tenantColumn)));
        } else {
            insert.getColumns().add(tenantColumn);
        }

        // 追加 tenant_id 值到每个 value 行（Long 类型，使用 LongValue 而非 StringValue）
        LongValue tenantValue = new LongValue(tenantId);
        ItemsList itemsList = insert.getItemsList();
        if (itemsList instanceof ExpressionList expressionList) {
            expressionList.getExpressions().add(tenantValue);
        } else if (itemsList instanceof MultiExpressionList multiExprList) {
            for (ExpressionList exprList : multiExprList.getExpressionLists()) {
                exprList.getExpressions().add(tenantValue);
            }
        }
        // INSERT ... SELECT：SELECT 子句中追加，并加入 column 列表
        if (insert.getSelect() != null) {
            PlainSelect ps = (PlainSelect) insert.getSelect().getSelectBody();
            ps.getSelectItems().add(new SelectExpressionItem(tenantColumn));
        }
    }

    private boolean shouldProcess() {
        Long tenantId = TenantContext.getTenantId();
        return tenantId != null;
    }

    private void addTenantCondition(Expression where, Statement statement) {
        Long tenantId = TenantContext.getTenantId();
        String tenantCol = properties.getTenantIdColumn();

        // 去重检测：如果 WHERE 中已存在 tenant_id = ? 条件，跳过追加
        if (where != null && containsTenantCondition(where, tenantCol)) {
            return;
        }

        EqualsTo tenantEquals = new EqualsTo();
        tenantEquals.setLeftExpression(new Column(tenantCol));
        tenantEquals.setRightExpression(new LongValue(tenantId));  // Long 类型

        if (where == null) {
            statement.setWhere(tenantEquals);
        } else {
            statement.setWhere(new AndExpression(where, tenantEquals));
        }
    }

    /**
     * 递归检测 WHERE 表达式中是否已包含 tenant_id 条件。
     * 避免重复追加导致 WHERE tenant_id = ? AND tenant_id = ?。
     */
    private boolean containsTenantCondition(Expression expr, String tenantCol) {
        if (expr instanceof EqualsTo eq) {
            if (eq.getLeftExpression() instanceof Column col
                && col.getColumnName().equals(tenantCol)) {
                return true;
            }
        }
        if (expr instanceof AndExpression and) {
            return containsTenantCondition(and.getLeftExpression(), tenantCol)
                || containsTenantCondition(and.getRightExpression(), tenantCol);
        }
        if (expr instanceof OrExpression or) {
            return containsTenantCondition(or.getLeftExpression(), tenantCol)
                || containsTenantCondition(or.getRightExpression(), tenantCol);
        }
        if (expr instanceof Parenthesis p) {
            return containsTenantCondition(p.getExpression(), tenantCol);
        }
        return false;
    }
}
```

**关键点**：
- 仅在 `TenantContext` 中存在租户 ID 时才注入。
- 忽略表直接跳过。
- SELECT/UPDATE/DELETE：无 WHERE 子句时新建条件，有则 AND 追加。
- INSERT：若 column 列表中无 `tenant_id`，自动追加列和值；已有则跳过（`MetaObjectHandler` 已填充）。
- INSERT ... SELECT：同时追加 column 和 SELECT 子句。
- 由于 SQL 改写发生在 MyBatis-Plus 的拦截器链中，Schema/Database 模式的切换已在 `TenancyStrategy.beforeSqlExecute` 中完成，此拦截器不会干扰。

> **Java 21+ 写法**：`insert.getColumns() instanceof ColumnList columnList && ...` 可直接解构，简化 `getColumns().add()` 链。

#### 3.4 与策略模块的协作

- 策略模块的 `beforeSqlExecute` 在拦截器链的**更早阶段**被调用（可通过自定义 `InnerInterceptor` 实现，或在 `beforePrepare` 中手动调用策略）。
- 或者，我们可以在上述 `TenantLineInnerInterceptor` 的 `beforePrepare` 中，先调用 `TenancyStrategyFactory.getStrategy(mode).beforeSqlExecute(invocation, tenantInfo)`，确保 Schema/数据源切换/表后缀设置完成后，再执行 SQL 解析改写。

**更优方案**：创建一个前置拦截器 `TenantStrategyInterceptor`，负责策略调度，顺序放在所有其他拦截器之前。

```java
public class TenantStrategyInterceptor implements InnerInterceptor {

    private final TenancyStrategyFactory strategyFactory;
    private final TenantInfoProvider tenantInfoProvider;

    public TenantStrategyInterceptor(TenancyStrategyFactory strategyFactory,
                                     TenantInfoProvider tenantInfoProvider) {
        this.strategyFactory = strategyFactory;
        this.tenantInfoProvider = tenantInfoProvider;
    }

    @Override
    public void beforePrepare(StatementHandler sh, Connection connection, Integer transactionTimeout) {
        Long tenantId = TenantContext.getTenantId();  // Long 类型
        if (tenantId == null) return;
        TenantInfo tenantInfo = tenantInfoProvider.getTenantInfo(tenantId);
        if (tenantInfo == null) {
            throw new TenantNotFoundException("Tenant not found: " + tenantId);
        }
        TenancyStrategy strategy = strategyFactory.getStrategy(tenantInfo.getMode());
        // 传递 Invocation 包装（需将当前上下文包装为 Invocation）
        Invocation invocation = new StatementHandlerInvocation(sh);
        strategy.beforeSqlExecute(invocation, tenantInfo);
    }
}
```

拦截器链顺序调整为：

```
1. BlockAttackInnerInterceptor
2. TenantStrategyInterceptor  <-- 新增，负责策略调度
3. DynamicTableNameInnerInterceptor
4. TenantLineInnerInterceptor
5. PaginationInnerInterceptor
```

### 4. 事务管理增强

#### 4.1 事务租户冻结

在事务开始时，通过 `TransactionSynchronizationManager` 注册同步器，记录并冻结租户。

```java
@Component
public class TenantTransactionSynchronization implements TransactionSynchronization {

    @Override
    public void afterCompletion(int status) {
        // 清理事务快照
        TransactionTenantHolder.clear();
    }
}

// 事务租户持有者
public class TransactionTenantHolder {
    private static final TransmittableThreadLocal<Long> TX_TENANT = new TransmittableThreadLocal<>();

    public static void freeze(Long tenantId) {
        TX_TENANT.set(tenantId);
    }

    public static Long get() {
        return TX_TENANT.get();
    }

    public static void clear() {
        TX_TENANT.remove();
    }
}
```

在事务切面（或通过自定义 `DataSourceTransactionManager` 包装）中冻结：

```java
@Aspect
@Component
public class TransactionalAspect {

    @Before("@annotation(org.springframework.transaction.annotation.Transactional)")
    public void freezeTenant() {
        Long tenantId = TenantContext.getTenantId();  // Long 类型
        if (tenantId != null) {
            TransactionTenantHolder.freeze(tenantId);
            TransactionSynchronizationManager.registerSynchronization(
                new TenantTransactionSynchronization());
        }
    }
}
```

#### 4.2 事务内切换检测

在 `TenantContext.runWithTenant` 的门面方法中增加检查：

```java
public static void runWithTenant(TenantPrincipal principal, Runnable task) {
    // 若已存在事务冻结租户且与目标不同，则拒绝
    Long frozenTenant = TransactionTenantHolder.get();
    if (frozenTenant != null && !frozenTenant.equals(principal.getTenantId())) {
        throw new TenantSwitchInTransactionException(
            "Cannot switch tenant from " + frozenTenant + " to " + principal.getTenantId() + " within a transaction");
    }
    holder.runWithTenant(principal, task);
}
```

#### 4.3 `@Transactional(propagation = REQUIRES_NEW)` 的租户继承

`REQUIRES_NEW` 会挂起当前事务、开启新事务和新连接。租户上下文的处理规则沿用 Spring 事务传播的语义：

| 传播级别 | 租户行为 | 原因 |
|---------|---------|------|
| `REQUIRED`（默认） | 继承外层租户 | 同一事务，同一连接 |
| `REQUIRES_NEW` | 继承外层租户 | 新事务但租户上下文不变——外层 `runWithTenant` 作用域未结束 |
| `NESTED` | 继承外层租户 | 同一连接的 savepoint |
| `SUPPORTS` | 继承外层租户 | 加入现有事务 |
| `NOT_SUPPORTED` | 继承外层租户 | 挂起事务但租户上下文仍在线程中 |

**关键原则**：租户上下文与事务无关——`TenantContext` 绑定的是线程（ScopedValue 作用域），不是事务。只要 `runWithTenant` 的作用域未结束，无论事务如何传播，租户上下文始终可见。

```java
@Transactional
public void outerMethod() {
    // 外层事务，租户 = tenantA
    TenantContext.getTenantId();  // → tenantA

    innerService.requiresNewMethod();  // 新事务，但租户仍是 tenantA
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void requiresNewMethod() {
    // 新事务 + 新连接，但租户上下文继承自外层
    TenantContext.getTenantId();  // → tenantA（继承）
    // 禁止在此方法内调用 runWithTenant(tenantB, ...)，否则抛出异常
}
```

**唯一例外**：如果 `REQUIRES_NEW` 方法内显式调用 `runWithTenant(tenantB, ...)`，`TenantSwitchInTransactionException` 会拦截——因为 `TransactionTenantHolder` 在**外层事务**已冻结 tenantA，新事务虽独立但仍在 `runWithTenant(tenantA)` 作用域内。

若需更全局的控制，可自定义 `DataSourceTransactionManager` 子类，在 `doBegin` 时冻结当前租户，在 `doCleanupAfterCompletion` 时清理。

### 5. 自动填充与连接清理

#### 5.1 TenantMetaObjectHandler

```java
@Component
public class TenantMetaObjectHandler implements MetaObjectHandler {

    @Override
    public void insertFill(MetaObject metaObject) {
        Long tenantId = TenantContext.getTenantId();  // Long 类型
        if (tenantId != null) {
            this.strictInsertFill(metaObject, "tenantId", Long.class, tenantId);
        } else {
            // 未开启租户功能时，tenant_id 默认为 0，不允许为 null
            this.strictInsertFill(metaObject, "tenantId", Long.class, 0L);
        }
    }

    @Override
    public void updateFill(MetaObject metaObject) {
        // 通常不更新租户字段
    }
}
```

> **未开启租户时的默认值规则**：当 `multi-tenancy.enabled = false` 或 `TenantContext.getTenantId() == null` 时，`tenant_id` 字段自动填充为 `0L`。**数据库中 `tenant_id` 字段不允许为 `NULL`**，始终设置 `DEFAULT 0`。这保证了无论是否开启租户功能，表结构保持一致，且未开启租户时所有数据归属于“平台默认租户”（tenantId = 0）。

> **DDL 约定**：所有业务表的 `tenant_id` 字段定义为 `BIGINT NOT NULL DEFAULT 0`。开启租户后由拦截器 / MetaObjectHandler 填充真实租户 ID；未开启时默认 0。

#### 5.2 连接清理拦截器

在策略执行后，SQL 执行完毕后，连接归还前需重置 Schema 环境。可通过自定义拦截器在 `afterCompletion` 中调用 `resetConnection`：

```java
public class ConnectionResetInterceptor implements InnerInterceptor {

    @Override
    public void afterQuery(Executor executor, MappedStatement ms, Object parameter,
                           RowBounds rowBounds, ResultHandler resultHandler, BoundSql boundSql) {
        resetIfNeeded();
    }

    @Override
    public void afterUpdate(Executor executor, MappedStatement ms, Object parameter) {
        resetIfNeeded();
    }

    private void resetIfNeeded() {
        // 从当前连接持有者获取连接，调用策略基类的 resetConnection
        // 实际实现时需获取到当前线程的连接，可通过 DataSourceUtils 等
    }
}
```

另一种方案：在动态数据源的 `getConnection()` 方法中包装连接，在 `close()` 时执行重置。

### 6. 数据源熔断与降级

> 对应 [可运维与灰度增强详细设计](可运维与灰度增强详细设计.md) §1 中的 `DataSourceCircuitBreaker`，本节定义熔断触发后的降级策略。

#### 6.0 熔断器 key 类型约定

`DataSourceCircuitBreaker` 内部使用 `Map<String, CircuitState>`，key 统一为 **String 类型**，取值规则如下：

| key 值 | 含义 |
|--------|------|
| `"shared"` | 共享数据源 |
| `String.valueOf(tenantId)` | 租户独立数据源（如 `"1001"`） |
| `TenantInfo.dataSourceName` | database 模式下的自定义数据源名（与上等价） |

所有调用 `circuitBreaker.isOpen(key)` / `recordFailure(key)` / `recordSuccess(key)` 的地方均使用此约定。`DynamicTenantDataSource` 的 lookup key 同样为 String，与熔断器 key 保持一致。

#### 6.1 故障模型

数据源故障按影响面分为两类，降级空间完全不同：

| 故障类型 | 影响范围 | 降级选项 | 恢复方式 |
|----------|----------|----------|----------|
| shared 数据源故障 | 全租户（column/table/schema 模式） | 无 fallback，全租户 503 | 健康探测恢复 |
| 独立 tenant 数据源故障 | 单租户（database 模式） | 仅该租户 503 | 健康探测恢复 |

> shared 数据源故障会同时影响 `sys_tenant` 元数据表的读写，因此即使 database 模式租户不受连接池影响，其元数据查询也会失败。这是 shared DS 故障影响面大于 tenant DS 故障的根本原因。

#### 6.2 熔断触发点

在 `TenantStrategyInterceptor.beforePrepare` 中，策略调度**之前**检查熔断状态：

```java
public class TenantStrategyInterceptor implements InnerInterceptor {

    private final DataSourceCircuitBreaker circuitBreaker;
    private final DataSourceHealthProbe healthProbe;

    @Override
    public void beforePrepare(StatementHandler sh, Connection connection,
                              Integer transactionTimeout) {
        // 0. 熔断检查（在策略调度之前）
        Long tenantId = TenantContext.getTenantId();
        if (tenantId != null && circuitBreaker.isOpen(String.valueOf(tenantId))) {
            throw new DataSourceUnavailableException(
                "Tenant data source is currently unavailable: " + tenantId);
        }
        // 同样检查 shared 数据源
        if (circuitBreaker.isOpen("shared")) {
            throw new DataSourceUnavailableException(
                "Shared data source is currently unavailable");
        }

        // 1. 策略调度（原有逻辑）
        TenantInfo tenantInfo = tenantInfoProvider.getTenantInfo(tenantId);
        TenancyStrategy strategy = strategyFactory.getStrategy(tenantInfo.getMode());
        strategy.beforeSqlExecute(...);
    }
}
```

#### 6.3 恢复策略

```java
/**
 * 由 DataSourceHealthProbe 定时调度（默认每 30s）。
 * 探测成功自动关闭熔断，零人工介入。
 */
public class DataSourceHealthProbe {

    private final DynamicTenantDataSource dynamicDataSource;
    private final DataSourceCircuitBreaker circuitBreaker;

    @Scheduled(fixedDelayString = "${multi-tenancy.health.probe-interval-ms:30000}")
    public void probeAll() {
        // 1. 探测 shared 数据源
        probeDataSource("shared", () -> dynamicDataSource.getResolvedDefaultDataSource());

        // 2. 探测所有 database 模式租户数据源
        for (Map.Entry<Object, DataSource> entry : dynamicDataSource.getResolvedDataSources().entrySet()) {
            if ("shared".equals(entry.getKey())) continue;
            probeDataSource(entry.getKey().toString(), entry.getValue()::getConnection);
        }
    }

    private void probeDataSource(String key, Supplier<Connection> connectionSupplier) {
        try (Connection conn = connectionSupplier.get()) {
            if (conn.isValid(3)) {
                circuitBreaker.recordSuccess(key);   // → HALF_OPEN → CLOSED
            } else {
                circuitBreaker.recordFailure(key);   // → 累计 failures
            }
        } catch (SQLException e) {
            circuitBreaker.recordFailure(key);
        }
    }
}
```

**状态机总览**：

```
CLOSED ──(failures ≥ threshold)──▶ OPEN
  ▲                                    │
  │                          (waitTime 超时)
  │                                    ▼
  └────(探测成功)──── HALF_OPEN ◀──────┘
         (探测失败)────▶ OPEN (重置计时)
```

- HALF_OPEN 期间仅放行健康探针的请求——业务请求仍然抛 `DataSourceUnavailableException`，避免洪峰瞬间涌入半恢复的数据源。
- 熔断关闭后，`DataSourceCircuitBreaker` 状态重置，租户恢复正常访问，无需运维介入。

### 7. 完整拦截器链组装顺序（最终版）

```
1. BlockAttackInnerInterceptor        // 防全表操作
2. TenantStrategyInterceptor          // 策略调度（Schema/DB/Table 预处理）
3. DynamicTableNameInnerInterceptor   // 表名替换（Table 模式）
4. TenantLineInnerInterceptor         // 列隔离条件注入（Column 模式）
5. PaginationInnerInterceptor         // 分页
6. ConnectionResetInterceptor         // 连接清理（Schema 重置）
```

在 `MybatisPlusInterceptor` 中按顺序添加以上拦截器，非适用模式由内部逻辑跳过。

### 8. 与上下文、策略模块的集成流程总结

1. **请求进入**：`TenantFilter` 绑定租户至 `TenantContext`。
2. **进入业务方法**：可能被 `@Transactional` 切面冻结事务租户。
3. **执行 Mapper 方法**：MyBatis-Plus 拦截器链触发。
4. `TenantStrategyInterceptor` 检查熔断状态 → 获取 `TenantInfo` → 调用对应策略：
   - Schema 策略：执行 `SET LOCAL search_path`，标记连接脏。
   - Database 策略：设置 `DataSourceContextHolder`，使 `DynamicTenantDataSource` 路由到租户库。
   - Table 策略：设置 `TableSuffixHolder`。
   - Column 策略：仅校验。
5. 后续拦截器根据上下文完成 SQL 改写：
   - `DynamicTableNameInnerInterceptor` 使用 `TableSuffixHolder` 替换表名。
   - `TenantLineInnerInterceptor` 使用 `TenantContext.getTenantId()` 注入列条件。
6. SQL 执行。
7. `ConnectionResetInterceptor` 在完成后重置连接 Schema（若适用）。
8. 事务结束后清理事务租户冻结。
9. 请求结束，`TenantContext` 自动清除。

### 9. 关键类图

- `DynamicTenantDataSource` 继承 `AbstractRoutingDataSource`，持有 `ConcurrentHashMap`。
- `TenantStrategyInterceptor` 依赖 `TenancyStrategyFactory`、`TenantInfoProvider`、`DataSourceCircuitBreaker`。
- `DataSourceCircuitBreaker` 状态机管理 shared + 各租户数据源的熔断状态，状态变更写入 changelog。
- `DataSourceHealthProbe` 定时探测数据源连接，联动 CircuitBreaker 实现自动恢复。
- `TenantTransactionSynchronization` 实现 Spring 的 `TransactionSynchronization`。
- `TenantMetaObjectHandler` 实现 `MetaObjectHandler`。

通过以上集成，持久层完整支持了多租户隔离，并保证了事务安全、连接清洁和故障自愈。
