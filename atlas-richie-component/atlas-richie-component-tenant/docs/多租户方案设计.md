# 基于 JDK 25 + Spring Boot 4.0.6 的多租户 MyBatis-Plus 通用插件设计方案

## 1. 文档目标

本文定义多租户插件的统一技术方案，目标如下：

- 通过配置实现多隔离模式切换（column / table / schema / database / hybrid）
- 在 Spring Boot 自动装配体系下实现低侵入接入
- 保证租户隔离安全、事务一致性与可运维性
- 预留 SPI 以支持企业级扩展（租户元数据、动态数据源、审计）

## 2. 方案范围与非目标

### 2.1 范围

- MyBatis-Plus 拦截器链租户增强
- DataSource 层 schema/database 路由
- 租户上下文传递与入口绑定机制
- 配置模型、默认实现与扩展 SPI

### 2.2 非目标

- 不解决跨系统统一身份治理（由 IAM/SSO 提供）
- 不默认实现分布式事务编排（仅预留集成点）
- 不支持同一事务内跨租户或跨库访问

## 3. 架构总览

采用“配置驱动 + 上下文 + 元数据 + 策略执行 + 持久层路由”分层模型：

1. **配置层**：定义隔离模式与资源参数  
2. **上下文层**：传递租户 ID（ScopedValue/ThreadLocal）  
3. **元数据层**：租户 → 隔离模式/数据源/schema 映射（SPI）  
4. **策略层**：column/table/schema/database 模式执行器  
5. **持久层**：MyBatis-Plus 拦截器链 + 动态 DataSource

## 4. 已确认架构决策（ADR）

### ADR-001：同一事务内禁止跨租户/跨库调用

- 决策：**不允许**
- 约束：
  - 事务开始即冻结 `tenantId`
  - 事务中若检测到租户切换，立即抛错并回滚
  - 禁止在 `@Transactional` 方法内部显式覆盖租户上下文
- 原因：避免越权与一致性破坏，降低排障复杂度

### ADR-002：混合模式按租户固定隔离模式

- 决策：混合模式采用 **按租户固定模式**
- 约束：
  - 同一租户的 `IsolationMode` 在一个发布周期内保持稳定
  - 模式变更走运维流程（元数据变更 + 灰度验证）
  - 不支持按业务域/表级混合路由
- 原因：降低 SQL 路由冲突、运维复杂度与测试矩阵规模

## 5. 配置模型

```yaml
multi-tenancy:
  enabled: true
  mode: column                    # column / table / schema / database / hybrid
  tenant-id-header: X-Tenant-ID  # 仅作为切换意图，不可单独信任
  enforce-auth-tenant: true       # 强制认证态租户归属校验
  tenant-id-column: tenant_id
  ignore-tables:
    - sys_config

  table-name-suffix: _${tenant}
  schema-prefix: tenant_

  datasource:
    shared:
      url: jdbc:postgresql://localhost:5432/shared_db
      username: app
      password: password
      hikari:
        maximum-pool-size: 20
    tenants:
      tenantA:
        url: jdbc:postgresql://localhost:5432/tenant_a
      tenantB:
        url: jdbc:postgresql://localhost:5432/tenant_b
```

**安全原则**：租户 ID 必须与认证主体（JWT/Session）绑定校验。Header 租户 ID 视为不可信输入。

## 6. 核心组件设计

### 6.1 TenantContext（上下文层）

```java
public final class TenantContext {
    private static final ScopedValue<TenantPrincipal> CURRENT_TENANT = ScopedValue.newInstance();

    public static void runWithTenant(TenantPrincipal principal, Runnable task) {
        ScopedValue.where(CURRENT_TENANT, principal).run(task);
    }

    public static <T> T runWithTenant(TenantPrincipal principal, Supplier<T> task) {
        return ScopedValue.where(CURRENT_TENANT, principal).call(() -> task.get());
    }

    public static Long getTenantId() {
        return CURRENT_TENANT.isBound() && CURRENT_TENANT.get() != null
            ? CURRENT_TENANT.get().getTenantId() : null;
    }

    public static TenantPrincipal get() {
        return CURRENT_TENANT.isBound() ? CURRENT_TENANT.get() : null;
    }
}
```

说明：

- 首选 `ScopedValue`（虚拟线程友好）
- 可提供 `ThreadLocal` 降级实现
- 跨线程池/MQ/定时任务必须在入口显式绑定上下文

### 6.2 TenantInfoProvider（元数据 SPI）

```java
public interface TenantInfoProvider {
    TenantInfo getTenantInfo(Long tenantId);
    boolean exists(Long tenantId);
}
```

`TenantInfo` 至少包含：

- `Long tenantId`
- `IsolationMode mode`
- `String dataSourceName`
- `String schemaName`
- `String tableSuffix`
- `TenantStatus status`
- `boolean canary`

### 6.3 TenancyStrategy（策略层）

```java
public interface TenancyStrategy {
    boolean supports(IsolationMode mode);
    void beforeSqlExecute(Invocation invocation, TenantInfo tenantInfo);
}
```

策略职责：

- `ColumnStrategy`：租户字段注入
- `TableStrategy`：动态表名替换
- `SchemaStrategy`：schema 切换
- `DatabaseStrategy`：数据源路由
- `HybridStrategy`：按租户固定模式分发

## 7. 关键实现规范

### 7.1 动态数据源路由

基于 `AbstractRoutingDataSource`：

- `determineCurrentLookupKey()` 根据 `TenantInfo` 决定 shared / tenant ds
- 动态新增数据源必须：
  1. 写锁保护更新映射
  2. `setTargetDataSources(newMap)`
  3. `afterPropertiesSet()` 刷新解析缓存
  4. 配套健康检查、上限与回收策略

### 7.2 Schema 切换安全

实现要求：

- schema 由白名单映射得到，不直接拼接 `tenantId`
- schema 标识符校验：`^[a-zA-Z0-9_]+$`
- 优先 `SET LOCAL search_path`（事务内生效，避免连接污染）
- 不支持 `SET LOCAL` 时，必须连接归还前 reset schema（或池级 resetSql）

### 7.3 MyBatis-Plus 拦截器链

推荐顺序：

1. DynamicTableNameInnerInterceptor
2. PaginationInnerInterceptor
3. TenantLineInnerInterceptor

写操作防护：

- INSERT 自动填充 `tenant_id`（MetaObjectHandler）
- UPDATE/DELETE 无租户条件时拒绝执行
- 启用全表更新/删除防护（BlockAttackInnerInterceptor）

**未开启租户时的默认值**：当 `multi-tenancy.enabled = false` 时，`tenant_id` 字段自动填充为 `0L`。数据库 `tenant_id` 字段定义为 `BIGINT NOT NULL DEFAULT 0`，不允许为 `NULL`。未开启租户时所有数据归属于“平台默认租户”（tenantId = 0）。

### 7.4 事务边界

- 租户上下文在入口层绑定（Filter/WebFilter/Listener）
- 事务内冻结租户，禁止切换
- 发现事务内切换租户立即抛错并回滚

## 8. 自动配置建议

```java
@Configuration
@ConditionalOnProperty(prefix = "multi-tenancy", name = "enabled", havingValue = "true", matchIfMissing = true)
@EnableConfigurationProperties(MultiTenancyProperties.class)
public class MultiTenancyAutoConfiguration {
    // TenantContext / TenantInfoProvider / 策略与拦截器 Bean 注册
}
```

建议通过 `AutoConfiguration.imports` 暴露自动配置。

## 9. 与 JDK 25 + Spring Boot 4.0.6 的结合点

- `ScopedValue`：替代 ThreadLocal，减少泄漏风险
- 虚拟线程：提升并发处理能力，仍需关注上下文入口绑定
- Spring Boot 自动配置：条件装配 + SPI 扩展
- GraalVM：动态代理/反射点需显式注册

## 10. 运行与运维基线

- 认证绑定：租户 ID 必须来自认证声明
- 审计字段：`tenantId/userId/sqlHash/datasource/schema/traceId`
- 数据源治理：容量上限、空闲回收、熔断与健康探测
- 压测重点：schema 切换、连接复用、事务并发、路由刷新
- 灰度策略：新租户/新模式先小流量验证再全量

## 11. 落地验收清单（DoD）

- [ ] 四种模式 + hybrid 在集成测试通过
- [ ] 事务内租户切换可被检测并回滚
- [ ] schema 切换无连接污染（压测验证）
- [ ] UPDATE/DELETE 无租户条件可被拦截
- [ ] 动态数据源增删与刷新线程安全
- [ ] 审计日志可追溯到租户与路由资源

## 12. 总结

该方案通过“配置 + 策略 + 路由 + 安全约束”构建统一多租户能力，并明确了事务与混合模式边界。按本文标准落地后，可在保证隔离安全的前提下兼顾扩展性与运维可控性。
