## 1. 数据源健康检查

### 1.1 模块定位

对 `DynamicTenantDataSource` 中所有数据源（shared + 各租户）进行连接池健康监控与熔断保护，通过 Actuator 端点暴露指标。

### 1.2 核心组件

#### DataSourceHealthIndicator

```java
@Component
public class DataSourceHealthIndicator implements HealthIndicator {

    private final DynamicTenantDataSource dynamicDataSource;

    public DataSourceHealthIndicator(DynamicTenantDataSource dynamicDataSource) {
        this.dynamicDataSource = dynamicDataSource;
    }

    @Override
    public Health health() {
        Health.Builder builder = Health.up();
        Map<String, DataSourceHealth> details = new LinkedHashMap<>();

        // 检查 shared 数据源
        DataSource shared = dynamicDataSource.getResolvedDefaultDataSource();
        details.put("shared", checkDataSource("shared", shared));

        // 检查所有租户数据源
        Map<String, DataSource> tenants = dynamicDataSource.getTenantDataSources();
        tenants.forEach((tenantId, ds) ->
            details.put(tenantId, checkDataSource(tenantId, ds)));

        // 任一数据源不健康则整体 DOWN
        boolean anyDown = details.values().stream()
            .anyMatch(h -> h.status() == DataSourceStatus.DOWN);
        if (anyDown) {
            builder.down();
        }

        return builder.withDetail("datasources", details).build();
    }

    private DataSourceHealth checkDataSource(String name, DataSource dataSource) {
        if (dataSource instanceof HikariDataSource hikari) {
            HikariPoolMXBean pool = hikari.getHikariPoolMXBean();
            if (pool == null) {
                return new DataSourceHealth(DataSourceStatus.UNKNOWN, 0, 0, 0, 0, -1);
            }
            int active = pool.getActiveConnections();
            int idle = pool.getIdleConnections();
            int pending = pool.getPendingConnections();

            // 判定健康：无挂起线程 且 活跃连接未打满
            DataSourceStatus status = (pending == 0 && active < pool.getMaximumPoolSize())
                ? DataSourceStatus.UP
                : DataSourceStatus.DEGRADED;

            return new DataSourceHealth(status, active, idle, pending,
                pool.getTotalConnections(), pool.getMaximumPoolSize());
        }
        // 非 HikariCP 数据源，尝试一次连接探测
        return probeConnection(name, dataSource);
    }

    private DataSourceHealth probeConnection(String name, DataSource ds) {
        try (Connection conn = ds.getConnection()) {
            boolean valid = conn.isValid(3);
            return valid
                ? new DataSourceHealth(DataSourceStatus.UP, -1, -1, -1, -1, -1)
                : new DataSourceHealth(DataSourceStatus.DOWN, -1, -1, -1, -1, -1);
        } catch (SQLException e) {
            return new DataSourceHealth(DataSourceStatus.DOWN, -1, -1, -1, -1, -1);
        }
    }
}

record DataSourceHealth(
    DataSourceStatus status,
    int activeConnections,
    int idleConnections,
    int pendingConnections,
    int totalConnections,
    int maxConnections
) {}

enum DataSourceStatus { UP, DEGRADED, DOWN, UNKNOWN }
```

#### 数据源熔断器

> 熔断后的降级策略、恢复流程、告警机制详见 [持久层路由与拦截器集成模块详细设计](持久层路由与拦截器集成模块详细设计.md) §6。

```java
/**
 * 当租户数据源连续失败超过阈值时，自动将其标记为"熔断"，
 * 后续请求直接返回 503（DataSourceUnavailableException），避免拖垮线程池。
 *
 * 状态机：CLOSED → (failures > threshold) → OPEN → (waitTime) → HALF_OPEN → (success) → CLOSED
 */
public class DataSourceCircuitBreaker {

    private final Map<String, CircuitState> states = new ConcurrentHashMap<>();
    private final int failureThreshold;      // 默认 5
    private final long openWindowMs;         // 默认 30_000

    public void recordFailure(String dataSourceKey) {
        states.compute(dataSourceKey, (k, state) -> {
            if (state == null) state = new CircuitState();
            state.failures++;
            if (state.failures >= failureThreshold) {
                state.status = CircuitStatus.OPEN;
                state.openedAt = System.currentTimeMillis();
            }
            return state;
        });
    }

    public boolean isOpen(String dataSourceKey) {
        CircuitState state = states.get(dataSourceKey);
        if (state == null || state.status != CircuitStatus.OPEN) return false;
        // 超时后尝试半开
        if (System.currentTimeMillis() - state.openedAt > openWindowMs) {
            state.status = CircuitStatus.HALF_OPEN;
            return false;
        }
        return true;
    }

    public void recordSuccess(String dataSourceKey) {
        CircuitState state = states.get(dataSourceKey);
        if (state != null && state.status == CircuitStatus.HALF_OPEN) {
            state.status = CircuitStatus.CLOSED;
        }
    }

    private static class CircuitState {
        int failures;
        CircuitStatus status = CircuitStatus.CLOSED;
        long openedAt;
    }
    enum CircuitStatus { CLOSED, OPEN, HALF_OPEN }
}
```

## 2. 灰度策略

### 2.1 模块定位

支持"新租户/新模式先小流量验证再全量"的灰度发布流程。灰度状态由 `TenantInfo.canary` 字段控制，在策略层和数据源路由层分别生效。

### 2.2 核心类型

#### TenantInfo 扩展

在现有 `TenantInfo` 基础上增加灰度字段（已纳入概念设计运行时对象模型）：

```java
public class TenantInfo {
    private Long tenantId;                  // Long 类型
    private IsolationMode mode;
    private String dataSourceName;         // 全量数据源
    private String canaryDataSourceName;   // 灰度数据源（与全量物理隔离）
    private String schemaName;
    private String tableSuffix;
    private TenantStatus status;           // ACTIVE / INACTIVE / MIGRATING
    private boolean canary;                // 当前是否处于灰度阶段
    // ...getters/setters
}
```

#### CanaryTenantManager

```java
/**
 * 灰度路由管理器。
 * 灰度租户列表与比例从 MultiTenancyProperties 读取，
 * Nacos 配置变更后 Spring 自动刷新 Bean，无需额外同步。
 */
@Component
public class CanaryTenantManager {

    private final MultiTenancyProperties properties;

    /**
     * 判断当前请求是否走灰度通道。
     * 每次调用实时读取最新配置（@RefreshScope 保证配置热更新）。
     */
    public boolean shouldRouteToCanary(Long tenantId, Object requestId) {
        MultiTenancyProperties.CanaryTenant canary = properties.getCanary().getTenants().stream()
            .filter(t -> t.getId().equals(tenantId))
            .findFirst().orElse(null);
        if (canary == null) return false;

        int ratio = canary.getRatio();
        if (ratio >= 100) return true;
        if (ratio <= 0) return false;

        // 一致性哈希：同一请求 ID 始终落同一通道
        int hash = Math.abs(Objects.hash(tenantId, requestId));
        return (hash % 100) < ratio;
    }
}
```

> 灰度状态的变更（进入/退出灰度、调整比例）通过修改 Nacos 中 `multi-tenancy.canary.tenants` 配置完成，`@RefreshScope` 自动生效。

### 2.3 与数据源路由的集成

灰度租户在 `DatabaseStrategy` 中需要路由到独立的**灰度数据源**，避免影响全量租户：

```java
// DatabaseStrategy.beforeSqlExecute 扩展
@Override
public void beforeSqlExecute(Invocation invocation, TenantInfo tenantInfo) {
    assertTenantPresent();
    validateTenantId(TenantContext.getTenantId());

    String dataSourceKey;
    if (tenantInfo.isCanary() && tenantInfo.getCanaryDataSourceName() != null) {
        // 灰度租户走独立数据源
        dataSourceKey = tenantInfo.getCanaryDataSourceName();
    } else {
        dataSourceKey = tenantInfo.getDataSourceName();
    }

    if (dataSourceKey == null) {
        throw new IllegalArgumentException("No datasource configured for tenant");
    }
    DataSourceContextHolder.set(dataSourceKey);
}
```

### 2.4 配置模型扩展

```yaml
multi-tenancy:
  datasource:
    tenants:
      1001:
        url: jdbc:postgresql://localhost:5432/tenant_1001
        canary-url: jdbc:postgresql://localhost:5432/tenant_1001_canary  # 灰度数据源

  canary:
    tenants:           # 灰度租户列表（可通过 Nacos 动态刷新）
      - id: 1001
        ratio: 30      # 30% 流量走灰度
      - id: 1002
        ratio: 100     # 100% 灰度（全量验证）
```

### 2.5 灰度流程

```
1. 运维在 Nacos 配置 multi-tenancy.canary.tenants 中添加租户，ratio=10
2. @RefreshScope 自动重建 MultiTenancyProperties Bean
3. CanaryTenantManager.shouldRouteToCanary() 实时读取新配置，10% 流量走 canary 数据源
4. 观察监控指标（错误率、延迟），无异常
5. 逐步提升 ratio：30 → 50 → 100
6. 100% 稳定运行 N 天后，运维从 Nacos 移除该租户的 canary 配置
7. 后续流量全部走原数据源，灰度数据源可下线
```

## 3. 动态配置刷新

### 3.1 设计原则

`MultiTenancyProperties` 本身标注 `@RefreshScope` + `@ConfigurationProperties`，Nacos 配置变更后 Spring Cloud 自动重建该 Bean。所有依赖配置的组件通过注入 `MultiTenancyProperties` 直接读取最新值，无需手动监听。

```java
@Component
@RefreshScope
@ConfigurationProperties(prefix = "multi-tenancy")
public class MultiTenancyProperties {

    // ===== 现有字段（来自主方案 §5） =====
    private boolean enabled;
    private IsolationMode mode;
    private boolean forceThreadLocal;
    private String tenantIdHeader;
    private boolean enforceAuthTenant;
    private String tenantIdColumn;
    private List<String> ignoreTables;
    private String tableNameSuffix;
    private String schemaPrefix;
    private DataSourceConfig datasource;

    // ===== 新增字段 =====
    private CanaryConfig canary = new CanaryConfig();
    private CircuitBreakerConfig circuit = new CircuitBreakerConfig();

    @Data
    public static class CanaryConfig {
        private List<CanaryTenant> tenants = new ArrayList<>();
    }

    @Data
    public static class CanaryTenant {
        private Long id;
        private int ratio = 100;
    }

    @Data
    public static class CircuitBreakerConfig {
        private int failureThreshold = 5;
        private long openWindowMs = 30_000;
    }

    // ... existing DataSourceConfig, SharedDataSourceConfig, TenantDataSourceConfig 等
}
```

### 3.2 支持动态刷新的配置

| 配置项 | 刷新方式 |
|--------|---------|
| `canary.tenants` | `CanaryTenantManager` 注入 `MultiTenancyProperties`，每次调用 `shouldRouteToCanary()` 时读取最新列表 |
| `circuit.failure-threshold` / `open-window-ms` | `DataSourceCircuitBreaker` 注入 `MultiTenancyProperties`，每次判断时读取最新阈值 |
| `ignore-tables` | 各拦截器注入 `MultiTenancyProperties`，每次 SQL 处理时读取最新列表 |

### 3.3 不支持动态刷新的配置

以下配置变更必须重启应用：

- `mode`（隔离模式）— 涉及拦截器链组装
- `tenant-id-column`（租户字段名）— SQL 改写逻辑依赖
- `force-thread-local` — JVM 级别，影响上下文实现选择

### 3.4 数据源增删

Nacos 中 `multi-tenancy.datasource.tenants` 配置变更后，`@RefreshScope` 重建 `MultiTenancyProperties`，同时触发 `RefreshScopeRefreshedEvent`。监听该事件完成数据源的增删重建：

```java
@Component
public class TenantDataSourceRefresher {

    private final DynamicTenantDataSource dynamicDataSource;
    private final DataSourceCircuitBreaker circuitBreaker;
    private final ReadWriteLock lock = new ReentrantReadWriteLock();

    @EventListener(RefreshScopeRefreshedEvent.class)
    public void onRefresh() {
        MultiTenancyProperties props = SpringContextHolder.getBean(MultiTenancyProperties.class);
        Map<String, MultiTenancyProperties.TenantDataSourceConfig> newTenants =
            props.getDatasource().getTenants();

        lock.writeLock().lock();
        try {
            Set<String> existingKeys = dynamicDataSource.getTenantDataSources().keySet();

            // 新增租户 → 创建数据源并注册
            newTenants.forEach((tenantId, config) -> {
                if (!existingKeys.contains(tenantId)) {
                    HikariDataSource ds = buildDataSource(config, props.getDatasource().getShared());
                    dynamicDataSource.addTenantDataSource(tenantId, ds);
                }
            });

            // 已删除租户 → 关闭连接池并移除
            existingKeys.forEach(tenantId -> {
                if (!newTenants.containsKey(tenantId)) {
                    DataSource removed = dynamicDataSource.removeTenantDataSource(tenantId);
                    if (removed instanceof HikariDataSource hikari) {
                        hikari.close();
                    }
                    circuitBreaker.remove(tenantId);
                }
            });
        } finally {
            lock.writeLock().unlock();
        }
    }
}
```
