## 1. KV / String 存储

### 设计思路

对 Redis String 数据结构做面向业务的封装，将 Redis 原生的 `SET/GET/INCR/DECR` 等命令包装为简洁的 Java 静态方法，同时集成 L2 本地缓存、布隆过滤器和性能守卫。

核心能力分层：

```
GlobalCache.value().set(k, v, ttl)
  ├── RedisPerfGuard.checkStringWritePayload()   ← 写入前检测反模式
  ├── L2 本地缓存写入（如启用）
  └── Redis SET
```

### 解决什么问题

- 业务代码直接操作 `RedisTemplate` 需要处理序列化、反序列化、key 拼接、异常处理等模板代码
- 多个开发者各自写 Redis 工具类导致代码散乱、用法不统一
- 无法统一做写入治理（大 Key 检测、String 反模式检测、慢查询监控）
- 数值、布尔类型需要反复序列化和反序列化

### 配置

```yaml
spring:
  data:
    redis:
      # 基础连接：继承 Spring Boot DataRedisProperties
      host: localhost
      port: 6379
      password:
      database: 0
      timeout: 3s

      # L2 二级缓存（读路径优先查本地缓存）
      enable-l2-caching: true
      l2-caching-data:
        - STRING          # String 类型走 L2
        - HASH

      # 性能守卫
      perf:
        enabled: true
        warn-string-payload-anti-patterns: true   # 检测 String 大值反模式
        string-payload-max-chars-warn: 100000     # 字符数 WARN 阈值
        string-payload-max-chars-error: 1000000   # 字符数 ERROR 阈值
        block-string-payload-violations: true     # 大值写入时抛异常阻断
```

### 推荐配置

```yaml
spring.data.redis:
  enable-l2-caching: true
  l2-caching-data: [STRING, HASH]
  perf:
    enabled: true
    warn-string-payload-anti-patterns: true
    block-string-payload-violations: true        # 生产推荐开启，阻止大 Key
```

### 设计收益

- **统一门面**：所有 String 操作经过 `GlobalCache`，不分散到各个业务代码
- **治理埋点**：天然的写入治理检测点，避免 Redis 大 Key
- **L2 加速**：开启 L2 后读路径优先走本地缓存，减少 Redis 网络开销
- **复杂度透明**：每方法标注时间复杂度，开发者调用时清楚性能影响

### 测试用例

```java
// 基础写读
GlobalCache.value().set("test:key", "hello", 60_000L);
String val = GlobalCache.value().get("test:key", String.class);
assert "hello".equals(val);

// 带过期时间的数值操作
GlobalCache.value().set("test:counter", 100, 60_000L);
long newVal = GlobalCache.value().increment("test:counter", 1L);
assert newVal == 101;

// 条件写入（仅不存在时写入）
boolean added = GlobalCache.value().setIfAbsent("test:token", "abc", 60_000L);
assert added;
boolean notAdded = GlobalCache.value().setIfAbsent("test:token", "xyz", 60_000L);
assert !notAdded;
```

## 2. Hash 操作

### 设计思路

封装 Redis Hash 数据结构的全量操作，支持：
- 单字段读写（`hset` / `hget`）
- 全对象存取（`putAll` / `entries`）
- 批量读取
- 带锁回源（防缓存击穿）

### 解决什么问题

- Hash 适合存储结构化对象，但每次读写整个对象序列化为 JSON 存入 String 是反模式（无法部分更新、BIGKEY 风险）
- 使用 Hash 存储对象字段可以做到按需读取、部分更新
- 防缓存击穿：高并发下大量请求同时读取不存在的 key → 全部穿透到 DB

### 配置

与 String 共用同一套 `spring.data.redis.*` 配置。Hash 操作特有的配置：

```yaml
spring.data.redis:
  l2-caching-data:
    - HASH               # Hash 类型启用 L2 本地缓存
  perf:
    warn-string-payload-anti-patterns: true  # 检测 Hash field 中塞入大对象
```

### 推荐配置

```yaml
spring.data.redis.enable-l2-caching: true
spring.data.redis.l2-caching-data: [STRING, HASH]
```

### 设计收益

- **对象友好**：`struct().set` / `struct().get` 自动序列化/反序列化 Java 对象
- **部分更新**：`field().set(key, field, value)` 只更新一个字段，不序列化整个对象
- **字段级 TTL**：每个 key 整体过期，配合 Hash 结构使用
- **防击穿**：`struct().getWithLock` 在缓存未命中时保证单一线程回源

### 测试用例

```java
// 单字段操作
GlobalCache.field().set("user:1", "name", "Tom");
GlobalCache.field().set("user:1", "age", 25);
String name = GlobalCache.field().get("user:1", "name", String.class);
assert "Tom".equals(name);

// 对象操作
UserInfo user = new UserInfo("Tom", 25);
GlobalCache.struct().set("user:2", user, 3600_000L);
UserInfo cached = GlobalCache.struct().get("user:2", UserInfo.class);
assert "Tom".equals(cached.getName());

// 防击穿回源
UserInfo result = GlobalCache.struct().getWithLock(
    "user:999", UserInfo.class, 3600_000L,
    () -> userRepository.findById("999")
);
```

## 3. 有界 List（`queue()` / `stack()`）

### 设计思路

原生无界 List API 已移除。List 场景统一通过 **`GlobalCache.queue()`**（FIFO 削峰）与 **`GlobalCache.stack()`**（LIFO 最近 N 条）提供容量治理（`maxLen` 1～4999）。

> **与 Stream MQ 的边界**：有界队列/栈均为**主动拉**、无消费组/ACK/死信，**不是**消息队列。可靠投递请用独立模块 **`StreamMQ`**（`atlas-richie-component-redis-streammq`）。详见 [§22](#22-有界队列-queue) / [§23](#23-有界栈-stack)。

### 解决什么问题

- 无界 List 在高并发下无限增长 → BIGKEY、阻塞、内存风险
- 需要可预期的削峰缓冲：队列满时淘汰队首（`queue`）或拒绝压入（`stack`）
- PaaS Redis 不支持 Stream 部分特性时的轻量异步通道（接受主动拉、至多一次语义）

### 配置

与 Redis 连接共用 `spring.data.redis.*`，无独立开关。容量由创建时 `maxLen` 决定。

### 推荐配置

```yaml
spring.data.redis:
  host: localhost
  port: 6379
  perf:
    enabled: true    # 生产建议开启，辅助发现异常 List 操作
```

### 设计收益

- **容量硬上限**：companion meta key + Lua 原子写路径
- **语义清晰**：JDK Queue / Stack 风格 API（`offer`/`poll`、`push`/`pop`）
- **批量拉取**：`drain(count)` 上限 20，控制单次 Redis 耗时

### 测试用例

```java
var queue = GlobalCache.queue().getOrCreate("task:pending", 500L, Task.class);
queue.offer(new Task("t1"));
Task t = queue.poll();
List<Task> batch = queue.drain(10);

var stack = GlobalCache.stack().create("audit:recent", 100L, Event.class);
stack.push(new Event("login"));
Event latest = stack.peek();
```

## 4. Set 操作

### 设计思路

封装 Redis Set 数据结构的操作，支持不重复集合的增删查、批量操作、随机弹出。

### 解决什么问题

- Set 天然去重，适合标签、黑白名单、在线用户等场景
- 需要判断元素是否存在（`existsInSet`）
- 随机弹出元素（`popDataFromSet`）— 抽奖、随机分配场景

### 配置

```yaml
spring.data.redis:
  l2-caching-data:
    - SET                # Set 类型启用 L2 本地缓存
```

### 推荐配置

```yaml
spring.data.redis.l2-caching-data: [STRING, HASH]
# SET 一般不推荐走 L2，因为 Set 通常数据量大，存本地内存反而占用 JVM 堆
```

### 设计收益

- 集合操作原子化：添加、删除、判断存在、交并差集
- 批量操作封装：批量添加、批量弹出，减少多次网络往返
- 与布隆过滤器联动：`getSetCacheWithLock` 防击穿

### 测试用例

```java
GlobalCache.collection().add("tags", "java");
GlobalCache.collection().add("tags", "redis");
boolean exists = GlobalCache.collection().exists("tags", "java");
assert exists;
Set<String> all = GlobalCache.collection().get("tags", String.class);
assert all.contains("redis");
```

## 5. ZSet 操作

### 设计思路

封装 Redis ZSet（有序集合）的全量操作，支持按分数排序、排名查询、范围查询、分值更新。

### 解决什么问题

- 排行榜场景：实时更新分数、查询 Top N、查询某成员的排名
- 优先级队列：按时间戳或权重排序
- 范围查询：按分数区间过滤

### 配置

ZSet 目前未在 `l2-caching-data` 的 `KeyTypeEnum` 中（KeyTypeEnum 只包含 STRING/HASH/LIST/SET）。ZSet 不走 L2 本地缓存。

### 推荐配置

无需额外配置。ZSet 数据量通常较大，不适合 L2 本地缓存。

### 设计收益

- **复杂度透明标注**：ZSet 大部分操作是 O(log n)，文档明确标注
- **排名查询**：`ranking().reverseRank` 封装 `ZREVRANK`
- **范围弹出**：`ranking().popMin` 支持批量弹出

### 测试用例

```java
GlobalCache.ranking().set("rank", "user1", 99.5);
GlobalCache.ranking().set("rank", "user2", 88.0);
long reverseRank = GlobalCache.ranking().reverseRank("rank", "user1");
assert reverseRank == 0;

Set<String> top = GlobalCache.ranking().range("rank", 0, 9, new TypeReference<String>(){});
assert top.contains("user1");
```

## 6. Key 管理

### 设计思路

封装 Redis Key 的元数据操作：删除、存在判断、过期时间设置、重命名、复制等。

### 解决什么问题

- 批量删除需要遍历键集合，原生 API 繁琐
- 过期时间操作需要手动处理时间单位转换
- 重命名操作有 `RENAMENX` 需求（不存在时才重命名）

### 配置

无特殊配置。Key 管理操作依赖基础 Redis 连接配置。

### 推荐配置

无需额外配置。

### 设计收益

- **批量删除封装**：`key().removeCache(Collection)` 一次调用批量删除
- **批量存在判断**：`key().countExistingKeys(Collection)` 返回存在的键数量
- **时间单位统一**：所有 TTL 参数统一使用毫秒，避免 `TimeUnit` 混用

### 测试用例

```java
GlobalCache.value().set("k1", "v1");
GlobalCache.value().set("k2", "v2");
boolean exists = GlobalCache.key().hasKey("k1");
assert exists;
long count = GlobalCache.key().countExistingKeys(List.of("k1", "k2", "k3"));
assert count == 2;
GlobalCache.key().removeCache(List.of("k1", "k2"));
assert !GlobalCache.key().hasKey("k1");
```

## 7. 批量操作

### 设计思路

将多次 Redis 操作合并为一次网络往返（pipeline 模式），降低批量场景下的延迟。

### 解决什么问题

- 逐 key 写入 N 个值 → N 次网络往返，延迟 = N × RTT
- pipeline 将 N 次操作合并为一次发送、一次接收

### 配置

```yaml
spring.data.redis:
  lettuce:
    pipeline:
      enabled: true  # 开启 Lettuce pipeline 支持
```

### 推荐配置

pipeline 默认通过 Lettuce 连接池自动启用批量特性，无需额外配置。

### 设计收益

- **批量写入**：`value().batchSet(Map)` 内部使用 pipeline 批量发送
- **批量读取**：`value().getMap(Collection, TypeReference)` 一次获取多个 key 的值
- **批量条件写入**：`value().batchSetIfAbsent(Map)` 结合 `SET NX` 批量操作

### 测试用例

```java
Map<String, String> data = Map.of("b1", "v1", "b2", "v2", "b3", "v3");
GlobalCache.value().batchSet(data, 3600_000L);
Map<String, String> values = GlobalCache.value().getMap(
    List.of("b1", "b2", "b3"),
    new TypeReference<String>(){}
);
assert values.size() == 3;
```

## 8. 分布式锁

### 设计思路

基于 Redisson FencedLock 的分布式锁实现，在 Redisson 之前额外增加一层 JVM 本地锁池（`CacheLockManager.LOCK_POOL`），形成"本地锁 → 可重入 → Redisson"三层获取顺序，支持乐观/悲观两种模式、锁续期、可重入。

**为什么需要本地锁池**：每一次 Redisson 的锁获取和释放都是一次 Redis 网络 RTT（约 1-5ms）。在同一个 JVM 内，如果有大量线程并发争抢同一个锁 key，在到达 Redisson 之前，这些线程的竞争完全可以在 JVM 内存中完成（约 0.01ms）。`CacheLockManager` 基于 `ConcurrentHashMap` 维护了一个本地锁注册表，记录当前 JVM 内哪些线程持有哪些锁。锁的获取和释放先查这个注册表——如果同 JVM 已有线程持有该锁，后续线程在本地就能感知到，不需要每次都穿透到 Redis。这个机制对业务代码完全透明，开启后不改变分布式锁的语义（最终仍然由 Redisson 保证跨实例互斥）。

**锁获取链路**：

```
optimisticLock / pessimisticLock(key, seconds)
  → GlobalCache.lock().optimistic* / pessimistic*
```

### 解决什么问题

- **分布式环境下的资源互斥访问**：跨 JVM 实例对同一资源的写操作必须互斥，Redisson FencedLock 提供 fencing token 机制保证分布式一致性
- **高争用场景下的 Redis 压力**：同一个 JVM 内数十个线程同时抢同一个锁 key，每一次都打 Redis RTT 会放大 Redis 的并发压力。本地锁池在 JVM 内完成首轮竞争，只有真正的"胜出者"才去请求 Redisson，避免了同一 JVM 内的无效网络开销
- **避免锁过期导致业务未完成时锁被释放**：`lockWithRenewal` 利用 Redisson 看门狗自动续期，业务处理时间超过锁租期时不会意外释放
- **同一线程可重入，避免死锁**：同一个线程重复获取同一把锁时，本地 `CacheLockManager` 直接识别并递增计数器，不走 Redis，同时避免自己锁自己

### 配置

```yaml
spring.data.redis:
  # 本地锁（JVM 锁池）—— 高争用场景推荐开启
  enable-local-lock: true

  # Redisson 配置（通过 redisson 自动配置）
  # 锁 TTL / 看门狗续期策略由 Redisson 自动管理
```

### 推荐配置

```yaml
spring.data.redis:
  enable-local-lock: true    # 已默认开启，高争用场景下显著降低 Redisson 压力
```

本地锁池默认已启用。绝大多数场景不需要关闭。仅在极端情况下（锁总量极大且命中率极低导致 `ConcurrentHashMap` 膨胀）可以考虑关闭，但在实际业务中几乎不会出现。

### 设计收益

- **三层锁降级**：本地锁 → 可重入 → Redisson，逐层降级减少 Redis 压力。本地锁池在 JVM 内解决首轮竞争——高争用时，同一 JVM 内大批线程不需要依次串行走 Redis RTT，在本地 `ConcurrentHashMap` 的 O(1) 查找阶段就感知到了锁状态
- **量化收益**：本地锁命中 ≈ 0.01ms（纯 JVM 内存），Redisson 锁获取 ≈ 1-5ms（一次网络 RTT）。100 个线程并发抢同一个锁时，本地锁池把 99 次网络 RTT 降为 0 次，只有真正拿到锁的线程会发起一次 Redisson 请求
- **锁续期**：`lockWithRenewal` 利用 Redisson 内置看门狗自动续期，防止业务未完成锁已过期
- **乐观/悲观双模式**：乐观锁 `tryLock(0, time)` 试一次即返回，不阻塞；悲观锁 `lock(time)` 阻塞直到超时，按需选择
- **AutoCloseable**：`CacheLock` 实现 `AutoCloseable`，try-with-resources 自动释放锁，杜绝忘记 unlock 导致的死锁
- **可重入检测**：同一线程重复获取同一把锁时，`tryReentrant` 通过 `CacheLockManager.existLock()` 在本地识别并递增计数器，不走 Redis 网络，同时确保重入计数准确

### 测试用例

```java
try (var lock = GlobalCache.lock().optimisticWithRenewal("lock:order:123", 5L)) {
    if (lock.success()) {
        orderService.processOrder("123");
    }
}

try (var lock = GlobalCache.lock().pessimistic("lock:order:456", 10L)) {
    if (lock.success()) {
        orderService.processOrder("456");
    }
}
```

## 9. 防缓存击穿（带锁读回源）

### 设计思路

为热点缓存读操作提供"L2 → 布隆过滤器 → Redis → 分布式锁 → DB 回源"的四层防护链路。核心方法是 `getXxxWithLock` 系列。

**完整链路**（以 `value().getWithLock` 为例）：

```
value().getWithLock(key, timeout, dbLoader)
  ├── 布隆过滤器 → Redis GET → 分布式锁 → DB 回源 → 回填缓存
```

### 解决什么问题

- **缓存穿透**：请求的 key 在 Redis 和 DB 中都不存在，每次请求都穿透到 DB
- **缓存击穿**：热点 key 过期瞬间，大量并发请求同时打到 DB
- 不阻塞所有请求：加锁失败的线程不会阻塞等待锁释放，而是重试读缓存（已被成功线程写入）

### 配置

```yaml
platform:
  cache:
    bloom-filter:                          # 布隆过滤器（防穿透）
      enable: true
      type: REDISSON                       # REDISSON / GUAVA
      expected-insertions: 10000000        # 预期插入量
      false-probability: 0.001             # 误判率

spring.data.redis:
  enable-l2-caching: true                  # L2 本地缓存
  l2-caching-data: [STRING, HASH]
```

### 推荐配置

```yaml
platform.cache.bloom-filter:
  enable: true                             # 数据量 > 1 万条的热点推荐开启
  type: REDISSON                           # 多实例推荐 REDISSON（跨进程共享）
  expected-insertions: 10000000
  false-probability: 0.001

spring.data.redis.enable-l2-caching: true
spring.data.redis.l2-caching-data: [STRING, HASH]
```

### 设计收益

- **四层防护**：L2 → 布隆 → Redis → 锁 + DB，逐层挡回，每一层命中就跳过后续
- **击穿零影响**：即使锁获取失败，也不会直接穿透到 DB（重试读已有缓存）
- **DB 自动回填**：回源数据自动写入缓存和布隆过滤器
- **统一方法**：`value()` / `struct()` / `field()` 均提供 `getWithLock`，使用模式一致

### 测试用例

```java
String value = GlobalCache.value().getWithLock(
    "config:app:pay_ratio", 3600_000L,
    () -> configService.getConfig("pay_ratio")
);

UserInfo user = GlobalCache.struct().getWithLock(
    "user:profile:123", UserInfo.class, 3600_000L,
    () -> userService.findById("123")
);

String name = GlobalCache.field().getWithLock(
    "user:profile:123", "name", String.class, 3600_000L,
    () -> userService.findById("123").getName()
);
```

## 10. L2 二级缓存

### 设计思路

在 Redis 之前增加一层 JVM 本地缓存（Caffeine / Ehcache / cache2k），形成 L1（本地）+ L2（Redis）+ DB 的三级缓存体系。

**读写路径**：

```
读：GlobalCache.value().get(key) / struct().get(key)
  ├── LocalCache.get(key)                    ← L1 本地
  ├── Redis GET                                 ← L2 Redis
  └── 未命中 → getWithLock 回源

写：GlobalCache.value().set(key, value, ttl)
  ├── Redis SET
  ├── LocalCache.put                            ← 双写 L1
  └── keyspace 通知 → 其他实例失效 L1
```

### 解决什么问题

- **缓存雪崩**：大量 key 同时过期 → 全部打到 Redis。L1 本地缓存分摊读压力
- **Redis 热点 key**：单个 key 被大量实例同时读 → Redis CPU 瓶颈。L1 本地命中直接返回
- **网络延迟**：每次读 Redis 都有网络往返。L1 命中 0 网络开销

### 配置

```yaml
spring.data.redis:
  # L2 总开关
  enable-l2-caching: true

  # 哪些数据类型走 L2（可选：STRING、HASH、LIST、SET）
  l2-caching-data:
    - STRING
    - HASH

  # L2 TTL 随机偏移（防雪崩）：1~10 分钟自动追加
```

### 推荐配置

```yaml
spring.data.redis:
  enable-l2-caching: true
  l2-caching-data:
    - STRING                          # 热点字符串推荐走 L2
    - HASH                            # 高频对象缓存推荐走 L2
    # LIST 和 SET 谨慎——数据量大的话本地内存压力大
```

**多实例一致性说明**：

L2 跨实例同步依赖 Redis 键空间通知（`notify-keyspace-events KEA`）：

```bash
# Redis 需配置（默认关闭，必须手动开启）
notify-keyspace-events KEA
```

- 实例 A 写入/删除 Redis key → Redis 触发 keyspace 事件 → 实例 B 收到通知 → 失效本地缓存
- **短暂脏读窗口**：从写入 Redis 到收到通知之间，实例 B 的本地缓存仍是旧值
- **最大容忍建议**：即使没有收到失效通知，L2 本地缓存的 TTL 也不应超过 `原始 TTL × 2`

### 设计收益

- **读性能提升**：L1 命中延迟 ≈ 0.1ms（JVM 内存），远低于 Redis 的 1-5ms
- **雪崩防护**：即使 Redis 大量 key 同时过期，L1 层仍有数据
- **跨实例同步**：`CacheSyncListener` 通过 Redis keyspace 通知保持各实例 L1 一致
- **精细化配置**：按数据类型（STRING/HASH/LIST/SET）分别控制是否走 L2

### 测试用例

```java
GlobalCache.value().set("l2:test", "value", 60_000L);
String val = GlobalCache.value().get("l2:test", String.class);
assert "value".equals(val);
```

## 11. 布隆过滤器

### 设计思路

提供布隆过滤器能力，与防缓存击穿方法联动，用于**高效判定 key 是否"一定不存在"**，从而拦截穿透到 DB 的无效查询。

支持两种实现：

| 实现 | 适用场景 | 特点 |
|------|---------|------|
| `GuavaBloomFilter` | 单实例 | JVM 内存，无网络开销，不可跨进程共享 |
| `RedissonBloomFilter` | 多实例 | 存在 Redis 中，跨进程共享，但每次判定有网络开销 |

### 解决什么问题

- 缓存穿透：大量请求查询不存在的 key（如被恶意攻击），布隆过滤器在 L1/L2 之前拦截
- 与 `value().getWithLock` 联动：布隆判定不存在则直接回源，跳过 Redis 查询和锁获取

### 配置

```yaml
platform:
  cache:
    bloom-filter:
      enable: false                   # 默认关闭，上线前评估后开启
      type: REDISSON                  # REDISSON（推荐多实例）/ GUAVA（单实例）
      expected-insertions: 10000000   # 预期插入元素数
      false-probability: 0.001        # 误判率（0.1%）
```

### 推荐配置

```yaml
platform.cache.bloom-filter:
  enable: true                        # 缓存穿透风险较高时推荐开启
  type: REDISSON                      # 生产环境多实例推荐
  expected-insertions: 10000000
  false-probability: 0.001
```

**开启建议**：
- 缓存数据量 > 1 万条且有穿透风险 → 开启
- 数据结构明确（如用户 ID、商品 ID 范围可控）→ 效果最佳
- 误判率 0.1% = 1000 次查询中有 1 次空回源，是可接受的范围

### 设计收益

- **零误伤拦截**：判定"不存在"就是 100% 不存在，直接拦截穿透查询
- **低误判率可配置**：通过 `false-probability` 控制空间和精度的 trade-off
- **与缓存联动**：写入缓存时自动更新布隆过滤器（`bloomFilter.add(key)`），保持一致性

### 测试用例

```java
String key = "test:bloom:exists";
GlobalCache.value().set(key, "value", 60_000L);

String val = GlobalCache.value().getWithLock(
    "test:bloom:notexists", 60_000L,
    () -> "loadedFromDB"
);
```

## 12. 性能守卫

### 设计思路

`RedisPerfGuard` 是一个运行时的 Redis 操作治理层，在所有 Manager 的每个 Redis 调用路径上，按照**复杂度分级 + 延迟阈值 + String 载荷检测**三个维度进行监控和可选阻断。

**治理维度**：

```
RedisPerfGuard.execute(manager, method, tier, supplier)
  │
  ├── 复杂度白名单校验 (tocAllowedComplexities)
  │     └── 不在白名单中 → ERROR + 可选阻断
  │
  ├── 非 O(1) WARN
  │     └── tier != O1 → warn log
  │
  ├── 慢查询分级
  │     ├── >= tocSoftMs → WARN
  │     └── >= tocHardMs → ERROR
  │
  └── String 写入反模式检测 (checkStringWritePayload)
        ├── 集合/Map/数组整包 → WARN
        ├── 超大文本 → ERROR + 可选阻断
        └── 疑似 JSON 整包 → WARN
```

### 解决什么问题

- 开发者无意中在 ToC 核心链路上调用 `KEYS *`、全量 `HGETALL` 等 O(n) 操作
- 大 Value 导致网络瓶颈和 JVM GC 压力，没有预警机制
- 慢查询没有分级告警，问题需要等用户投诉才发现

### 配置

```yaml
spring.data.redis:
  perf:
    enabled: false                              # 默认关闭，灰度开启
    warn-non-o1: true                          # 非 O(1) 操作打 WARN
    toc-soft-ms: 8                             # 软阈值（毫秒），超过 WARN
    toc-hard-ms: 50                            # 硬阈值（毫秒），超过 ERROR
    block-forbidden-tiers: false               # 阻断 LINEAR_N / WORSE 操作
    toc-allowed-complexities:                  # ToC 允许的复杂度白名单（空=不限制）
      - O1
      - LOG_N
      - SCRIPT_OR_UNKNOWN
    warn-string-payload-anti-patterns: true    # 检测 String 反模式
    block-string-payload-violations: false     # 阻断 String 反模式写入
    string-payload-max-chars-warn: 100000      # String 字符数 WARN
    string-payload-max-chars-error: 1000000    # String 字符数 ERROR
```

### 推荐配置

**灰度阶段**：

```yaml
spring.data.redis.perf:
  enabled: true
  warn-non-o1: true
  toc-soft-ms: 8
  toc-hard-ms: 50
  block-forbidden-tiers: false       # 灰度阶段只告警不阻断
  warn-string-payload-anti-patterns: true
  block-string-payload-violations: false
```

**生产阶段**（消除告警后）：

```yaml
spring.data.redis.perf:
  enabled: true
  block-forbidden-tiers: true        # 阻断 O(n) / O(n²) 操作
  block-string-payload-violations: true  # 阻断大 Value 写入
```

### 设计收益

- **复杂度治理落地**：不是停留在文档里说"不要用 KEYS"，而是运行时检测到就 WARN 甚至阻断
- **性能劣化可观测**：`toc-soft-ms / toc-hard-ms` 自动检测慢查询，输出 `[RedisPerf]` 日志
- **BIGKEY 预防**：`checkStringWritePayload` 在写入前检测集合、Map、超大文本的 String 存储反模式
- **渐进式灰度**：`enabled` 先 false → 灰度 true（只告警）→ 生产 true（阻断），避免一次性影响

### 测试用例

```java
// 测试慢查询检测（启用 perf）
// 手动设置 toc-soft-ms: 1，然后执行一个需要 2ms 的操作
// 预期输出：[RedisPerf] latency soft threshold RedisStringManager setValue O1 2ms (>=1ms)

// 测试 String 反模式检测
// 写入一个 1MB 大小的 String
String bigValue = "x".repeat(2_000_000);
// 预期：block-string-payload-violations=true 时抛出 IllegalStateException
try {
    GlobalCache.value().set("test:big", bigValue, 60_000L);
    if (perf.blockStringPayloadViolations) {
        fail("should have been blocked");
    }
} catch (IllegalStateException e) {
    assert e.getMessage().contains("String value anti-pattern");
}

// 测试复杂度阻断
// block-forbidden-tiers=true 时，调用 O(n) 操作预期被阻断
```

## 13. 多 Redis 实例路由

### 设计思路

通过 `MultiRedisTemplate` 和 `MultiStringRedisTemplate` 支持**命名子 Redis 实例**，根据 key 前缀或业务名称路由到不同的 Redis 节点。

### 解决什么问题

- 不同业务隔离到不同 Redis 实例（如订单 Redis、用户 Redis）
- 读写分离：主 Redis 写、从 Redis 读
- 多环境共享：一个应用连接多个 Redis 集群

### 配置

```yaml
spring.data.redis:
  # 主 Redis 配置
  host: primary.redis.com
  port: 6379

  # 命名子 Redis 实例
  slaves:
    order-redis:
      host: order.redis.com
      port: 6379
      password: order-pass
    user-cache:
      host: user.redis.com
      port: 6380
```

### 推荐配置

```yaml
# 需要多实例隔离时启用
spring.data.redis.slaves:
  order-redis:
    host: ${ORDER_REDIS_HOST}
    port: ${ORDER_REDIS_PORT}
  user-cache:
    host: ${USER_REDIS_HOST}
    port: ${USER_REDIS_PORT}
```

### 设计收益

- **命名路由**：通过 key 前缀或业务名路由到指定 Redis，无需硬编码
- **共享连接池**：主 Redis 和子 Redis 各自独立连接池，互不影响
- **透明切换**：业务代码无感知，路由逻辑封装在 `MultiRedisTemplate` 中

### 注意

`MultiRedisTemplate` 的路由表在配置时静态初始化。运行时新增/删除子 Redis 实例需要重启。

## 15. 限流

### 设计思路

基于 Redis + Lua 脚本的滑动窗口限流算法。通过 `limiter().tryAcquire(key, maxCount, windowSeconds)` 判断请求是否在窗口内超过阈值。

### 解决什么问题

- 接口防刷
- API 限流（如每个用户每分钟最多调用 100 次）
- 突发流量控制

### 配置

无需额外限流配置。限流参数在调用时按业务传入：

```java
// 60 秒窗口内最多 100 次
boolean pass = GlobalCache.limiter().tryAcquire("rl:/api/pay", 100, 60);
```

### 推荐配置

限流参数应由业务方根据接口容量评估设置。推荐的窗口和阈值：

```yaml
# 建议在配置中心动态管理限流阈值（不硬编码在代码中）
rate-limiter:
  /api/pay: { max-count: 100, window-seconds: 60 }
  /api/order: { max-count: 50, window-seconds: 60 }
  /api/login: { max-count: 5, window-seconds: 60 }
```

### 设计收益

- **轻量级**：单条 Lua 脚本实现，无需额外依赖
- **分布式生效**：基于 Redis，所有实例共享同一限流状态
- **Lua 原子性**：`INCR + EXPIRE` 在 Lua 脚本中原子执行，无竞态条件
- **性能守卫集成**：限流调用经过 `RedisPerfGuard` 治理

### 测试用例

```java
String limitKey = "rl:test:api";
// 窗口内最多 5 次
for (int i = 0; i < 5; i++) {
    assert GlobalCache.limiter().tryAcquire(limitKey, 5, 60);
}
assert !GlobalCache.limiter().tryAcquire(limitKey, 5, 60);
```

## 19. 本地缓存（独立于 L2）

### 设计思路

提供独立于 Redis 的 JVM 本地缓存能力（基于 JSR-107 标准，支持 Caffeine / Ehcache / cache2k）。与 L2 不同，这个本地缓存是**业务显式使用**的独立区域（`spring.data.local`），而 L2 是 `GlobalCache` 读路径上自动通过的透明加速层。

### 解决什么问题

- 不需要 Redis 的纯本地缓存场景（如配置项、路由表）
- L2 是按数据类型自动启用的，业务开发者可能需要独立的命名缓存区域
- 通过 JSR-107 标准接口兼容多种底层实现

### 配置

```yaml
spring.data.local:
  provider: CAFFEINE      # CAFFEINE / EHCACHE / CACHE2K
  cache-definitions:
    - name: localConfig
      max-size: 10000
      expiry-policy: CREATED
      ttl: 600_000        # 10 分钟
    - name: routeTable
      max-size: 5000
      expiry-policy: ACCESSED
      ttl: 3_600_000      # 1 小时
```

### 推荐配置

```yaml
spring.data.local:
  provider: CAFFEINE              # Caffeine 性能最优
  cache-definitions:
    # 配置缓存：短 TTL、小容量
    - name: appConfig
      max-size: 1000
      expiry-policy: CREATED
      ttl: 300_000
    # 路由表缓存：中等 TTL、中等容量
    - name: routeTable
      max-size: 5000
      expiry-policy: ACCESSED
      ttl: 3_600_000
```

### 设计收益

- **标准接口**：基于 JSR-107（`javax.cache.Cache`），实现可切换
- **独立于 Redis**：不需要 Redis 连接即可使用本地缓存
- **多区域隔离**：通过 `CacheName` 区分不同缓存区域，各自的容量和 TTL 相互独立

## 20. 发布订阅 / 键空间通知

### 设计思路

封装 Redis Pub/Sub 发布订阅能力，主要用于两个场景：

1. **键空间通知 (Keyspace Notification)**：`CacheSyncListener` 订阅 Redis 的 `__keyevent@*__` 事件，用于多实例 L2 缓存同步；业务自定义监听用 `GlobalCache.event().subscribeKeyEvent(pattern, listener)`
2. **业务 Topic 发布订阅**：轻量广播用 `GlobalCache.notification().publish(topic, message)`

### 解决什么问题

- 多实例部署下 L2 本地缓存的一致性同步
- 轻量级的业务消息广播（不需要 Stream MQ 的持久化和 ACK 保障时）

### 配置

```yaml
# Redis 必须开启键空间通知（L2 多实例同步依赖）
# Redis 配置（redis.conf）：
notify-keyspace-events KEA
```

### 推荐配置

```bash
# Redis 服务端配置（所有环境）
notify-keyspace-events KEA
# K = keyspace 事件
# E = keyevent 事件
# A = 所有类型的通知（del、set、expired 等）
```

### 设计收益

- **L2 一致性保障**：`CacheSyncListener` 接收其他实例的写/删操作通知，自动失效本地缓存
- **轻量级广播**：不需要持久化的消息场景可以使用 Pub/Sub，比 Stream 更轻量

### 注意

Pub/Sub 是"即发即忘"模式，如果消费者不在线，消息会丢失。需要可靠消息的场景应使用 **`StreamMQ`**（`atlas-richie-component-redis-streammq`）。

## 21. 迁移窗口验证

### 设计思路

`MigrationWindowValidator` 在 Spring 启动时扫描 `AtlasRedisProperties` 中带有 `@MigrationWindow` 注解的字段，检查是否已过截止日期。如果过期且值仍为 `false`，则输出 ERROR 日志并阻止应用启动。

### 解决什么问题

- 配置项默认值往往是最"安全"的（比如 `perf.enabled: false`），但长期不开启就无法发挥价值
- "软开关永远没人改"的问题——通过到期熔断强制团队推进配置迁移

### 配置

被 `@MigrationWindow` 标注的字段：

```yaml
spring.data.redis.perf:
  enabled: true                           # 截止日期：2026-12-01
  block-forbidden-tiers: true             # 截止日期：2026-12-01
  block-string-payload-violations: true    # 截止日期：2026-12-01
```

### 推荐配置

```yaml
spring.data.redis.perf:
  enabled: true
  block-forbidden-tiers: true
  block-string-payload-violations: true
```

### 设计收益

- **强制执行**：到期后不移除配置直接启动失败，确保配置治理不会遗留
- **灰度平滑**：到期之前可以逐步灰度（先告警后阻断），不强制一刀切
- **异常逃生**：到期后发现启动失败 → 运维立即感知 → 1 分钟内配置上线

### 注意

生产环境服务如果启动失败影响较大，应该在截止日期**之前**完成灰度验证和配置迁移，而不是等启动失败再去改。

## 22. 有界队列 queue()

### 设计思路

在 Redis List 之上提供**有界 FIFO 队列**（`BoundedQueue`），通过 companion meta key（`{key}:meta`）持久化 `maxLen`，写路径 Lua 原子读取 meta 并维护 List 长度。相比已移除的无界 List API，核心是**容量治理 + 语义清晰的 JDK Queue 风格 API**。

**定位（重要）**：

- **主动拉**：业务须调用 `poll()` / `drain(count)` 消费，不拉不消费，无推送、无消费组
- **削峰缓冲工具**：高峰 `offer` 暂存，Worker 按能力拉取；满时**淘汰队首**（丢最老），换取系统不被拖死
- **轻量退路**：云 PaaS Redis 不支持 Stream 部分特性、又不想上 RocketMQ/RabbitMQ 时的 List 方案
- **不是 MQ**：可靠投递、ACK、死信、消费监控请用 **`StreamMQ`**（`atlas-richie-component-redis-streammq`）

**Redis 数据模型**：

```
{queueKey}        → Redis LIST（元素序列）
{queueKey}:meta   → String，存 maxLen（1～4999）
```

**写路径（offer）**：

```
BoundedQueue.offer(item)
  ├── assertMetaPresent()                    ← meta 存在性
  └── Lua: GET meta → RPUSH → LTRIM -maxLen,-1   ← 原子入队 + 裁剪
```

**扩容（grow）**：

```
grow() / queue().grow(key)
  ├── Lua GROW_MAX_LEN（×2，封顶 4999）
  ├── 队列实例：成功后 TRIM list 对齐 meta
  └── 已达封顶返回 false；meta 不存在/非法抛异常
```

**读路径与反序列化**：

```
poll / peek / peekTail / drain
  └── BoundedListElementConverter
        ├── Redis 无元素 → 静默 null（非错误）
        └── 反序列化失败 → WARN + null（poll 已弹出则数据已从 List 移除）
```

**核心类**：`BoundedQueueFunction` → `RedisBoundedQueueManager` → `BoundedQueue`；共用 `BoundedListRedisSupport`、`BoundedListRedisScripts`、`BoundedListCapacityLimits`。

### 解决什么问题

- 原生 List API **无 maxLen**，高并发下 List 无限增长 → BIGKEY、阻塞、内存风险
- `addListItem` + `leftPop` 语义分散，与「缓存整表 List」混用，团队难以统一治理
- 需要**可预期的削峰**：过载时丢最老任务，而不是拖垮 Redis 或 JVM
- PaaS 环境 Stream 能力受限时，仍需**基于现有 Redis** 的轻量异步通道（接受主动拉、至多一次语义）
- 多实例 `getOrCreate` 时 maxLen 不一致、与 `rawList()` 共 key、非 List 类型占 key 等**误用**需在创建期拦截

### 配置

与 List 共用 `spring.data.redis.*` 连接配置，**无独立 `queue.*` 开关**。容量由创建时 `maxLen` 与平台常量约束：

| 常量 | 值 | 含义 |
|------|-----|------|
| `LIST_BIGKEY_RECOMMENDED_MAX_ELEMENTS` | 5000 | README List 推荐业务上限 |
| `BOUNDED_MAX_LEN_CEILING` | 4999 | `maxLen` 合法上限（严格低于 BIGKEY 红线） |
| `MIN_MAX_LEN` | 1 | 最小容量 |
| `MAX_BATCH_COUNT` | 20 | `drain(count)` 单次批量上限 |

```yaml
spring.data.redis:
  host: localhost
  port: 6379
  # 有界队列无额外开关；依赖 JSON 序列化（jsonTemplate）
```

### 推荐配置

- **maxLen**：按峰值缓冲量评估，默认从小容量创建（如 500～2000），仅必要时 `grow()` 单次 ×2，避免一步顶到 4999
- **消费侧**：定时任务或专用 Worker **主动** `poll` / `drain(≤20)`；多实例竞争消费时自行评估「抢到即消费」是否可接受
- **key 规范**：`{业务}:bq:{场景}:{id}`，**禁止**与 `rawList()`、Stream key、String 缓存共 key
- **TTL**：短生命周期缓冲可 `expire(ms)`（list + meta Lua 双 key 同时 PEXPIRE）
- **类型稳定**：元素 `Class<T>` 与写入时序列化格式一致，避免反序列化 WARN 与静默丢条

### 设计收益

- **容量硬顶**：平台封顶 4999，对齐 List BIGKEY 治理，避免无界 List
- **原子写**：`offer` 单 Lua（RPUSH + LTRIM），并发下长度与 meta 一致
- **削峰语义明确**：满则丢队首，适合「宁可丢旧任务也不拖死」的场景
- **创建期校验**：`TYPE` 仅允许 `none`/`list`；`setIfAbsent` 仅 `TRUE` 算成功；`getOrCreate` 校验 maxLen 一致
- **可观测**：读路径反序列化失败统一 WARN（含 key、operation、failure）
- **与 Stream 分层清晰**：不抢占 Stream 的「真正 MQ」定位，降低误用成本

### 测试用例

```java
// 创建与入队出队
BoundedQueue<Task> q = GlobalCache.queue().create("order:bq:retry", 1000, Task.class);
q.offer(new Task("t1"));
q.offer(new Task("t2"));
Task first = q.poll();
assert "t1".equals(first.getId());

// 批量 drain（count 须在 1～20）
List<Task> batch = q.drain(10);

// getOrCreate：已存在时 maxLen 必须一致
BoundedQueue<Task> same = GlobalCache.queue().getOrCreate("order:bq:retry", 1000, Task.class);

// 扩容：仅 ×2，封顶 4999；不存在抛 IllegalArgumentException
boolean grown = GlobalCache.queue().grow("order:bq:retry");

// 勿与 rawList 共 key；非 List 类型 key 创建时抛 IllegalStateException
```

## 23. 有界栈 stack()

### 设计思路

在 Redis List 之上提供**有界 LIFO 栈**（`BoundedStack`），与队列共用 meta key 模式（`{key}:meta`）及 `BoundedListCapacityLimits` / `BoundedListRedisScripts` / `BoundedListRedisSupport`，但**溢出策略与消费语义不同**。

**定位**：

- 同为有界 List **工具**，**主动拉**（`pop` / `latest`），**不是**消息队列
- 适合「只关心最近 N 条」的 LIFO 场景（最近操作记录、最新一批待处理项预览等）
- 与 `queue()` 对比：栈 **满则拒绝 push**（返回 `false`），不自动淘汰旧元素

**Redis 数据模型**（与队列相同）：

```
{stackKey}        → Redis LIST（栈底→栈顶顺序与 RPUSH 一致，栈顶在右侧）
{stackKey}:meta   → String maxLen
```

**写路径（push）**：

```
BoundedStack.push(item)
  └── Lua: GET meta → LLEN >= maxLen ? return 0 : RPUSH
        └── 返回 1 成功，0 表示已满拒绝
```

**读路径**：

```
pop()      → RIGHTPOP（栈顶弹出）
peek()     → LINDEX -1
latest(n)  → LRANGE -n,-1（n ∈ [1,20]，只读不删）
```

**扩容**：仅更新 meta（`grow` Lua），**不**像队列那样 grow 后强制 LTRIM（栈未满时 LLEN ≤ maxLen 自然成立）。

**核心类**：`BoundedStackFunction` → `RedisBoundedStackManager` → `BoundedStack`。

### 解决什么问题

- 需要**固定深度**的 LIFO 结构，且满时希望**显式失败**而非静默覆盖（与队列「丢最老」对比）
- 原生 `rightPush` / `rightPop` 无 maxLen，无法表达「最多保留 N 条最近记录」
- `latest(count)` 只读批量窥探栈顶，避免为看一眼栈顶而 `pop` 破坏数据

### 配置

与 [§22 有界队列](#22-有界队列-queue) 相同：无独立配置项，`maxLen` 受 `BoundedListCapacityLimits` 约束。

### 推荐配置

- **场景选型**：仅当业务语义是 LIFO 时用 `stack()`；FIFO 削峰用 `queue()`
- **maxLen**：通常小于队列（如 50～500），表示「最近 N 条」窗口
- **满栈处理**：`push` 返回 `false` 时业务应降级（丢弃本次、告警或先 `grow()` 再试）
- **latest**：预览用 `latest(count)`，真正消费用 `pop()`；`count` 不超过 20

### 设计收益

- **满则拒绝**：保护已在栈内的 N 条数据不被隐式覆盖，语义可预测
- **栈顶 O(1)**：`peek` / `pop` / `push` 均为 List 端点操作
- **共用治理基建**：meta、grow、expire、destroy、类型检查、反序列化 WARN 与队列一致
- **API 与 JDK Deque 对齐**：降低从本地栈迁移到分布式栈的学习成本

### 测试用例

```java
BoundedStack<Event> stack = GlobalCache.stack().create("audit:stack:svc1", 100, Event.class);
boolean ok = stack.push(new Event("e1"));
assert ok;
Event top = stack.peek();
assert "e1".equals(top.getName());

// 满栈拒绝
for (int i = 0; i < 200; i++) {
    stack.push(new Event("fill-" + i));
}
boolean rejected = stack.push(new Event("overflow"));
assert !rejected;

// 批量窥探栈顶（不删除）
List<Event> recent = stack.latest(5);

Event popped = stack.pop();
```

### queue() 与 stack() 选型对照

| 维度 | `queue()` FIFO | `stack()` LIFO |
|------|----------------|----------------|
| 溢出策略 | 丢最老（LTRIM） | 满则 `push` 失败 |
| 典型场景 | 削峰、轻量任务缓冲 | 最近 N 条记录、栈式处理 |
| 消费方式 | `poll` / `drain` | `pop` / `latest` |
| 与 Stream MQ | 互补，非替代 | 互补，非替代 |
| 主动拉 | 是 | 是 |
