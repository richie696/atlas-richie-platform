# 设计模式速查表

常见业务场景与适合的设计模式对应起来，防止在 Service 中写出大量 if/else/switch 分支。

## 1. 按枚举 / 标识分支的差异逻辑

**场景**：支付方式、渠道类型、业务类型、地区策略等。

**推荐模式**：
- **策略模式（Strategy）**：不同实现封装为不同策略
- **模板方法（Template Method）**：主流程固定，部分步骤在子类实现
- **命令模式（Command）**：把操作封装为对象，便于排队/回放/审计
- **外观模式（Facade）**：对外只暴露一个统一接口，内部再分派策略/命令

## 2. 规则多且经常变

**场景**：优惠规则、积分规则、风控规则、额度规则、审批规则等。

**推荐模式**：
- **职责链模式（Chain of Responsibility）**：规则串行校验/过滤
- **策略模式 + 组合模式（Composite）**：规则可按 AND/OR 等组合
- 复杂时可考虑规则引擎或 DSL（解释器模式）

## 3. 多步骤流程与状态驱动

**场景**：订单生命周期、审批流程、开户流程、结算流程。

**推荐模式**：
- **模板方法**：主流程固定，步骤可定制
- **状态模式（State）**：每个状态一个对象，封装在该状态下的行为和允许的状态转移
- **策略 + 状态**：状态决定可用策略集合

> 注意：对于复杂状态流转，优先使用 `atlas-richie-component-statemachine` 组件，而不是在 Service 里堆 if/else。

## 4. 渠道 / 租户 / 产品族差异

**场景**：toC/toB 行为差异；多渠道（APP/H5/第三方）策略不同；不同产品/套餐产生不同行为。

**推荐模式**：
- **策略模式**：根据渠道/租户选择不同策略
- **装饰器模式（Decorator）**：在基础策略上叠加渠道/租户特有行为
- **抽象工厂（Abstract Factory）**：为不同渠道/租户创建对应"对象族"

## 5. 多下游调用并发 + 聚合

**场景**：聚合接口，需并发调用多个下游服务，然后汇总结果。

**推荐模式**：
- **结构化并发 + Facade/Orchestrator + 命令/策略**
- Orchestrator 负责拆任务、并发执行、聚合
- 单个任务实现可用命令/策略封装

> 参考 `docs/concurrency-patterns.md` 中的高 TPS 接口编码流程。

## 6. 可补偿的跨服务长事务

**场景**：下单 + 支付 + 库存预占 + 通知，需要支持部分失败补偿。

**推荐模式**：
- **命令模式 + 补偿命令（Saga/TCC 思路）**
- **状态模式**：记录当前步骤状态、支持恢复/重试

## 7. 横切能力（日志、审计、限流）

**场景**：在现有逻辑基础上按需叠加能力。

**推荐模式**：
- **装饰器模式**：按需包装基础服务
- **代理模式 + AOP**：使用框架级动态代理统一处理

## 8. 读写分离与查询模型

**场景**：写逻辑与读取模型差异大、查询复杂多变。

**推荐思想**：
- **CQRS**：命令（写）和查询（读）分离
- 为查询部分设计专门的 QueryService，可内部再用策略/组合模式

## 9. 兜底与防御

**场景**：防止业务代码传入非法参数、过大 payload、无界查询。

**推荐模式**：
- **Guard / Inspector**：在平台层（`perf/` 目录）加防御逻辑，如 `RedisPerfGuard`、`RedisStringPayloadInspector`
- **不要只在 Service 里加 if 判断**——放到平台层，统一生效

> 这是 `atlas-richie-platform` 的核心设计哲学：**为业务代码做兜底和下限，防止业务代码无下限**。
